# Phase 5 ŌĆö UI Design Contract

> Visual and interaction contract for Fatigue Model & Plateau Detection. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase delivers **decision-driving fatigue surfaces** on the Today tab: per-muscle volume bars with verb labels and a two-tone direct/indirect fill, a front/back body silhouette heatmap drillable to per-muscle detail, a plateau stall flag and suggested-action chip on `SessionExerciseCard`, a dismissible deload advisory banner that stacks above the existing `BlockCard`, a `MuscleVolumeTargetEditor` in Settings, per-exercise plateau threshold overrides in `ExerciseDetailView`, and a weekly recap sheet that auto-surfaces at the first app open of a new Monday.

The design stance is identical to Phases 1, 2, 3, and 4 verbatim: **"comprehensive but uncluttered"** ŌĆö progressive disclosure is the tie-breaker when density and clutter conflict; every new component stays on iOS-native chrome; no custom design system is introduced. This UI-SPEC carries forward every token from Phases 1ŌĆō4 verbatim: accent `#0E7C86` / `#3FBFC9`, the 8-point spacing scale, the 4-role / 2-weight semantic typography, the 60/30/10 color split, and the terse-prescriptive copywriting voice. Anything below that does not restate a prior-phase token is a Phase 5 addition.

**Exceptions (load-bearing for Phase 5):**

- **44pt minimum touch target** carries forward for all tappable controls. Specifically: each tappable muscle region in `BodySilhouetteView` ŌĆö the `.contentShape(scaledPath)` modifier ensures the hit area matches the anatomical path, but very small regions (neck, forearms) must be padded to a minimum 44pt ├Ś 44pt clickable area via `.frame(minWidth: 44, minHeight: 44)` on the Path view; the `DeloadAdvisoryBanner` dismiss button (`xmark` SF Symbol) has a 44pt hit area via `.contentShape(Rectangle()).frame(minWidth: 44, minHeight: 44)`; all `Stepper` controls in `MuscleVolumeTargetStepper` use system-provided hit areas (Ōēź44pt by system default).
- **Volume bar track height:** 16pt visual height (consistent with a compact information-dense display; matches the Phase 3 warm-up ramp bar height if established ŌĆö otherwise 16pt is the floor for legibility at this scale). - **Stall badge on `SessionExerciseCard`:** rendered as a 24pt capsule (label-only, no leading icon to save space) positioned trailing of the exercise name, left of the chevron; the suggested-action chip below it is 32pt high with 8pt horizontal padding inside. ### Volume zone color palette (Phase 5 additions ŌĆö Asset Catalog entries)

The four `VolumeZone` cases map to four semantic color assets in `Assets.xcassets`. Using Asset Catalog entries (the same approach as Phase 2's `PinnedNoteYellow`) ensures consistent dark-mode adaptation and avoids `Color.red` literals.

| VolumeZone | Asset name | Light hex | Dark hex | Usage |
|------------|-----------|-----------|----------|-------|
| `.belowMEV` | `Color("VolumeBelowMEVGray")` | `#8E8E93` | `#636366` | Bar fill, heatmap tint when below MEV |
| `.productive` | `Color("VolumeProductiveGreen")` | `#34C759` | `#30D158` | Bar fill, heatmap tint when in productive range (MEV..<MAV) |
| `.nearMRV` | `Color("VolumeNearMRVAmber")` | `#FF9500` | `#FF9F0A` | Bar fill, heatmap tint when near MRV (MAV..<MRV); also used for `DeloadAdvisoryBanner` accent border and bar ring tint while advisory is active |
| `.overMRV` | `Color("VolumeOverMRVRed")` | `#FF3B30` | `#FF453A` | Bar fill, heatmap tint when over MRV; pulse animation source color |

**Saturation curve:** Each zone uses a single flat color at full saturation for the direct-set (primary) fill. The indirect-set (secondary) fill uses the same color at `opacity(0.40)`. No gradient within a zone ŌĆö discrete zone colors keep the encoding consistent with the heatmap. The user learns one palette and applies it everywhere.

**OverMRV red treatment:** The `.overMRV` zone uses `Color("VolumeOverMRVRed")` at full saturation ŌĆö the same system red family as the Destructive role. This intentional overlap communicates urgency ("you are exceeding recoverable volume ŌĆö this is a problem"). An optional pulse animation (see Interaction & Motion section) draws additional attention. Full red saturation is used rather than a softened tint because the "Over MRV ŌĆö deload recommended." signal warrants the highest visual urgency in the zone hierarchy.

**DeloadAdvisory amber accent:** While a `DeloadAdvisory` is active and not dismissed, volume bars receive a subtle amber ring ŌĆö a 2pt `RoundedRectangle` stroke in `Color("VolumeNearMRVAmber").opacity(0.6)` drawn over the bar track. This communicates "the fatigue signal is active" without obscuring the zone fill. The ring appears on ALL bars when the advisory is active, not just bars in the nearMRV zone.

**Asset implementation:** These four entries are added to `fitbod/Assets.xcassets/` in individual `.colorset` directories following the same JSON structure as the existing `AccentColor.colorset` and `PinnedNoteYellow.colorset` (Phase 2 precedent). Light/dark mode values declared directly ŌĆö no system-semantic adaptation needed since these are data-visualization semantic colors, not UI chrome.

**Explicitly NOT accent in Phase 5:**

- **Volume bar fill colors:** Use the four Asset Catalog zone colors (gray/green/amber/red), not accent. Volume bars communicate zone state, not a CTA.
- **Heatmap muscle region fills:** Same zone colors, not accent.
- **Stall badge itself (`PlateauSignal.stalled`):** Uses `Color(.systemOrange)` capsule (consistent with the Phase 2 set-type chip "warning" semantic ŌĆö orange = "attention needed"). The SUGGESTED ACTION chip below it uses accent.
- **`DeloadAdvisoryBanner` background:** `Color("VolumeNearMRVAmber").opacity(0.15)` ŌĆö amber, not accent. The banner is an advisory signal, not an action prompt. Matches Phase 4's `ConsiderDeloadBanner` yellow convention but using amber to align with the VolumeZone amber (the advisory fires from similar fatigue signals that drive nearMRV coloring on bars).
- **`DeloadAdvisoryBanner` dismiss button:** `.secondaryLabel` foreground on `xmark`, not accent. Dismissal is secondary to reading the signal.
- **`DeloadAdvisoryBanner` "Tap to see signals" affordance:** System default button styling (`.body .systemBlue` ŌĆö the standard iOS link/button styling). Not accent ŌĆö the advisory detail sheet is navigation, not a primary action.
- **Heatmap section header ("Muscle Heatmap") and `FatigueSurfaceView` section headers:** `.headline .label`, never accent. Section headers are never accent per Phase 1 convention.
- **Volume bar verb labels and delta lines:** `.secondaryLabel` foreground. These are data, not CTAs.

**D-07 reconciliation note:** CONTEXT.md D-07 describes the bar fill style as "solid accent fill for direct (role == primary) weighted-set contribution + lighter accent for indirect." That wording was superseded in this UI-SPEC by the zone-color implementation. D-07's "accent fill" intent is preserved as a two-tone fill pattern (direct at full opacity + indirect at `opacity(0.40)`), but the color source changes from accent to zone color. Volume bars communicate zone state, not a CTA ŌĆö they correctly do NOT use the app accent color. The two-tone direct/indirect encoding from D-07 is fully retained.

## Interaction & Motion

### Tap targets (all Ōēź 44pt per iOS HIG)

- Every `MuscleVolumeBar` row: tappable via `.onTapGesture` on the VStack ŌĆö the entire bar card is the tap target (no explicit `.frame(minHeight: 44)` needed since the VStack with 4 rows naturally exceeds 44pt; verify via runtime inspection).
- Each `BodySilhouetteView` muscle region: `.contentShape(scaledPath)` on the scaled `Path` ensures hit-testing is precise to the anatomical shape. Very small regions (neck, forearms) are padded: the `Path` View has `.frame(minWidth: 44, minHeight: 44)`.
- `DeloadAdvisoryBanner` dismiss (`xmark`) button: `.contentShape(Rectangle()).frame(minWidth: 44, minHeight: 44)`.
- `StallBadge` + `SuggestedActionChip`: the chip is the tap target; `.frame(minHeight: 44)`.
- All `MuscleVolumeTargetStepper` Stepper controls: system `Stepper` provides Ōēź44pt hit area by default.

### Navigation transitions

- Tapping a `MuscleVolumeBar` row or a `BodySilhouetteView` muscle region ŌåÆ `NavigationLink(value: muscleSlug)` push onto the Today tab's `NavigationStack`. `PerMuscleDetailView` uses standard push transition.
- `DeloadAdvisoryBanner` tap on primary text ŌåÆ `DeloadSignalDetailSheet` presented as `.sheet` with `[.medium, .large]` detents (medium default ŌĆö the signal list is typically 1ŌĆō3 rows).
- `SuggestedActionChip` tap for `.addVolume` ŌåÆ push `PerMuscleDetailView` (same NavigationLink).
- `SuggestedActionChip` tap for `.dropIntensity` ŌåÆ push Phase 3's `PrescriptionEditorRow` (existing Phase 3 surface).
- `SuggestedActionChip` tap for `.deload` ŌåÆ present `DeloadSignalDetailSheet` as `.sheet`.
- `SuggestedActionChip` tap for `.tryVariation` ŌåÆ present `TryVariationSheet` as `.sheet` with `[.medium, .large]` detents.
- `WeeklyRecapSheet` ŌåÆ presented via `.sheet(isPresented: $showWeeklyRecap)` with `.presentationDetents([.large])` from `RootView`'s Today tab `.task`.
- Settings "Volume Targets" row ŌåÆ `NavigationLink` push `MuscleVolumeTargetEditor`.

### Dismiss gestures

- `DeloadAdvisoryBanner`: `xmark` tap (the dismiss gesture; calls `onDismiss` closure which sets `UserSettings.deloadAdvisoryDismissedWeekStart = currentMonday`). No swipe-to-dismiss on the banner itself (it is a top banner rendered via `.safeAreaInset`, not a sheet).
- `DeloadSignalDetailSheet` / `TryVariationSheet`: system sheet swipe-down gesture + "Done" toolbar button both dismiss.
- `WeeklyRecapSheet`: "Done" toolbar button only ŌĆö `.interactiveDismissDisabled(false)` (swipe-down works). No "Snooze" ŌĆö "Done" and swipe-down are both equivalent (the `weeklyRecapShownForWeekStart` guard already prevents re-surfacing).

### Pulse animation on `.overMRV`

When a `MuscleVolumeBar` enters the `.overMRV` zone, the direct-fill `RoundedRectangle` receives a repeating pulse animation:

```swift
// Applied conditionally when zone == .overMRV
.opacity(overMRVPulse ? 1.0 : 0.7)
.animation(
    zone == .overMRV
        ? .easeInOut(duration: 1.2).repeatForever(autoreverses: true)
        : .default,
    value: zone
)
@State private var overMRVPulse: Bool = false
// .onAppear: overMRVPulse = (zone == .overMRV)
// .onChange(of: zone): overMRVPulse = (zone == .overMRV)
```

The pulse oscillates the fill opacity between 1.0 and 0.7 over 1.2 seconds, repeating indefinitely while the zone is `.overMRV`. This draws attention without being garish ŌĆö the bar's color is already a strong red signal; the pulse adds temporal urgency.

**Reduced motion compliance:** The pulse animation is gated on `@Environment(\.accessibilityReduceMotion)`:

```swift
.animation(
    (zone == .overMRV && !reduceMotion)
        ? .easeInOut(duration: 1.2).repeatForever(autoreverses: true)
        : .default,
    value: zone
)
```

When Reduce Motion is ON, the fill stays at full opacity (no pulsing); the color is still the full red signal.

### Heatmap region tint

Each `BodySilhouetteView` muscle region `Path` is filled with a single flat zone color (no gradient within a zone ŌĆö consistent with the volume bars). There is no saturation curve based on set count within a zone: the discrete 4-zone encoding is the only encoding. Partial fills would introduce a 5th visual variable (saturation) that is not represented anywhere else in the system and would conflict with the zone-color contract.

### `BodySilhouetteView` wave strategy

- **Wave 0 (stub):** Each muscle region in `MuscleRegionPaths.front` and `MuscleRegionPaths.back` is a `Path` that draws a simple `Rectangle` at an approximate body-region position. Hit-testing works; visual fidelity is placeholder. Tests for "registry contains all 17 slugs" pass.
- **Wave 3 (anatomical refinement):** Each `Path` is replaced with anatomically-correct bezier curves hand-traced from a public-domain body reference (Wikimedia Commons CC-0 anatomy silhouette). The `MuscleRegionPathsTests` suite requires the slug registry to contain all 17 muscles regardless of wave; it does not assert path geometry (which is manually verified per VALIDATION.md).
