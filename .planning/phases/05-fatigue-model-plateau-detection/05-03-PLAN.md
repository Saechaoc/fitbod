<objective>
Ship the **Slice D vertical**: a serious lifter opens an active session and sees per-exercise stall flags + a single auto-picked suggested-action chip. PlateauDetector pure function fires PROG-06 closure end-to-end (window/tolerance/intent-split/per-exercise-override/high-rep-suppression all honored per D-10..D-13). The TryVariationSheet provides the Branch 4 fallback discovery surface; the per-exercise plateau threshold override editor in ExerciseDetailView closes SET-06.

Purpose: this is the FIRST plateau-aware surface the user sees during a workout. The chip on the exercise card is the bridge between Phase 5's analytical primitives (PlateauDetector) and the user's decision-driving moment ("my e1RM has been flat for 4 sessions — what should I do about it?"). The single-chip auto-pick (D-13) is the planner's anti-information-overload bet — instead of asking the user to interpret raw e1RM range data, the heuristic picks the most likely productive next action and surfaces it as a tappable affordance.

Output: 1 new pure-function service (`PlateauDetector`), 3 new SwiftUI components (`StallBadge`, `SuggestedActionChip`, `TryVariationSheet`), 2 modified Phase 1/2 views (`SessionExerciseCard`, `ExerciseDetailView`), 2 GREEN unit test suites (PlateauDetector + SuggestedAction; replacing 2 of the 8 RED scaffolds from plan 01), 1 GREEN UI test suite (PlateauOverride).

End-to-end user-visible deliverable: with a test-seeded session containing 4 historical sessions of barbell bench press all within ±2% top-set e1RM AND chest volume below MAV this week, the executor opens the active session and sees the bench press exercise card carrying a 'Stalled' badge plus an 'Add volume' chip. Tapping the chip navigates to the (plan 05) PerMuscleDetailView. Separately, opening ExerciseLibrary → 'Barbell Bench Press' → 'Plateau Detection' section, the user can toggle 'Override global settings' ON, change Window to 6 sessions, change Tolerance to 3.5%, and the values round-trip via SwiftData.
</objective>
