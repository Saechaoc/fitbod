# Phase 6 — UI Design Contract

> Visual and interaction contract for Progress Views, Export & Polish. Generated by gsd-ui-researcher (autonomous mode). Pre-populated from `06-CONTEXT.md` (D-01..D-37), `06-RESEARCH.md` (Patterns 1–6, Pitfalls 1–13, Schema-reality delta), and Phase 3 UI-SPEC (token carry-forward). Only the four items marked "Claude's Discretion" in `06-CONTEXT.md` were decided in this document (chart styling, filter-chip form factor, `ProgressHomeView` layout, banner copy variations); every other field is anchored upstream.

This phase delivers **the resolution at which serious lifters actually inspect their training** plus **data ownership**: a new Progress tab with per-exercise time-series charts, a PRs view, a weekly tonnage chart, a session-comparison view, an in-session live PR banner overlay on `SessionLoggerView`, CSV/JSON/`.fitbodbackup` export, and two-step destructive restore. **No new design tokens.** Every Phase 6 surface composes Phase 1/2/3 tokens verbatim.

The design stance is identical to Phases 1–3: **"comprehensive but uncluttered"** — progressive disclosure when density and clutter conflict. Native iOS chrome only. Swift Charts is the sole chart framework (no DGCharts, no custom Canvas, no third-party SPM).

## Spacing Scale

**Carries forward verbatim from Phases 1–3.** Multiples of 4, iOS 8-point grid.

| Token | Value | Phase 6 usage |
|-------|-------|---------------|
| xs | 4pt | Gap between PR-banner middot separators (`·`) and chips; gap between filter-chip dot indicator and label; spacing between Δ value and Δ unit in `SessionComparisonView` (e.g. "+5" — "kg") |
| sm | 8pt | Vertical gap between filter-chip rows in `WeeklyTonnageView`; gap between chart frame and the series-toggle chip row in `ExerciseProgressView`; padding inside the in-session PR banner capsule |
| md | 12pt | Vertical padding inside `ExercisePRsView` PR rows; row vertical padding in `SessionComparisonView`; chart inset above/below frame in `ExerciseProgressView` |
| lg | 16pt | Default horizontal inset for every new view (`ProgressHomeView`, `ExerciseProgressView`, `ExercisePRsView`, `WeeklyTonnageView`, `SessionComparisonView`); horizontal inset for the in-session PR banner (edge-to-edge minus 16pt); primary content margin in Settings → Data section |
| xl | 24pt | Section spacing in `ProgressHomeView` between the "Per-exercise" cards block and the "Weekly Tonnage" / "PRs" / "Compare to last week" entry rows; vertical spacing between the chart and the toggle-chip row in `ExerciseProgressView` |
| 2xl | 32pt | Empty-state vertical padding on every new view |
| 3xl | 48pt | (unchanged — top spacer above empty-state heading on tall device) |

**Exceptions (load-bearing for Phase 6):**

- **44pt minimum touch target** carries forward for all tappable controls. Specifically: every filter chip in `WeeklyTonnageView` (`[Last 8 wk]`, `[Last 26 wk]`, `[All time]`, block-phase chips, muscle chips), every series-toggle chip in `ExerciseProgressView` (`[Top set]`, `[All-set avg]`), and the PR row tap targets in `ExercisePRsView` must have a `.contentShape(Rectangle()).frame(minHeight: 44)` hit area. Chip visual height is ~32pt (matches Phase 3 chip-style precedent), but the tappable region extends to 44pt vertically via padding.
- **PR banner capsule height:** 44pt tall — intentionally touch-target height so the user can tap-to-dismiss with one tap. The banner is full edge-to-edge minus 16pt horizontal inset (lg).
- **Chart frame minimum height:** `ExerciseProgressView` chart `minHeight: 240` (per RESEARCH Pattern 1); `WeeklyTonnageView` chart `minHeight: 280` (slightly taller for `BarMark` legibility when stacked).
- **Per-exercise PR row height:** 56pt (two-line rendering: weight × reps on row 1, date + RPE on row 2). 12pt vertical padding × 2 + ~32pt content = ~56pt total — exceeds 44pt touch target.
- **PR banner chip gap:** Chips inside the banner are separated by a centered middot `·` glyph with 4pt horizontal padding on each side (xs token). No extra background per chip — the whole banner is one capsule.
- **Filter-chip row layout — horizontal scroll, NOT wrap.** Each filter row in `WeeklyTonnageView` is a `ScrollView(.horizontal, showsIndicators: false)` containing an `HStack(spacing: 8pt)` of chips. **Rationale:** Wrapping would cause unpredictable row heights as more muscle chips are added (top-10 trained muscles can vary by user state); horizontal scroll keeps each row 44pt tall and the chart frame's vertical position stable. The first chip in each row gets 16pt leading padding (lg); the last chip gets 16pt trailing padding so the user can flick-scroll without bumping the edge.

### Accent reserved for (Phase 6 surfaces; appended to the Phase 1 + Phase 2 + Phase 3 list)

The accent appears **only** at the following elements in Phase 6, in addition to all Phase 1, 2, and 3 surfaces already listed. Every other interactive control uses default system styling.

Phase 1 surfaces (still reserved): items 1–7 from `01-UI-SPEC.md`.
Phase 2 surfaces (still reserved): items 8–17 from `02-UI-SPEC.md`.
Phase 3 surfaces (still reserved): items 18–21 from `03-UI-SPEC.md`.

Phase 6 additions:

22. **Top-set chart series** in `ExerciseProgressView` — `LineMark` + `PointMark` use the project accent `#0E7C86`. This is the primary series; it conveys the lifter's session-level "best work." Stroke width: 2pt solid for strength sessions, 2pt dashed (`StrokeStyle(lineWidth: 2, dash: [4, 3])`) for hypertrophy sessions. `PointMark` symbol size 60.
23. **Weekly tonnage bar fill** in `WeeklyTonnageView` — `BarMark` foreground style is `Color.accentColor` (the project accent) when no muscle filter is active (single-color bars). When the muscle filter is active and the chart switches to stacked-bar mode, individual muscle segments use the **system color palette** (`.foregroundStyle(by: .value("Muscle", …))` lets Swift Charts assign colors from its built-in scale) — the accent is reserved for the unstacked baseline only.
24. **"Series" toggle chips** in `ExerciseProgressView` (`[Top set]` `[All-set avg]`) — selected state: `Color.accentColor.opacity(0.15)` fill + `Color.accentColor` foreground label. Unselected state: `Color(.systemGray5)` fill + `Color(.secondaryLabel)` foreground label. Matches Phase 3 filter-chip styling (single visual source for all chip-style controls).
25. **PR banner capsule** in `SessionLoggerView` (`InSessionPRBanner`) — background uses `Color.accentColor.opacity(0.15)` fill with `Color(.label)` text and a 1pt `Color.accentColor.opacity(0.3)` border. Chip separator middots (`·`) use `Color(.secondaryLabel)`. Rationale: the banner is a **success/affirmation state** mid-session, not a primary CTA — the soft tinted accent reads as "noteworthy, non-disruptive" and visually distinguishes from the Phase 3 bump banner (which uses `Color(.systemGreen).opacity(0.15)`). The two banners coexist in different scrolling layers (Phase 3 bump banner is per-exercise-card inline; Phase 6 PR banner is `.safeAreaInset(edge: .top)` at the view root).
26. **Filter chips** in `WeeklyTonnageView` (time range, block phase, muscle) — selected state uses the same chip style as #24 (accent-tinted). Unselected uses the same Phase 3 chip style (systemGray5 + secondaryLabel). The block-phase chip row is **disabled** with `.opacity(0.5)` and a footer caption "No block data yet — define a block in Routines" until Phase 4 ships (D-19).
27. **PR row Δ value highlight** in `SessionComparisonView` — when the Δ is positive (improvement), the Δ value renders in `Color.accentColor` `.headline` weight. When the Δ is negative (regression), it renders in `Color(.secondaryLabel)` `.headline` (NOT `.systemRed` — a lower session isn't an error, it's data). When the Δ is exactly zero, it renders in `Color(.tertiaryLabel)` (the neutral "no change" reading is the lowest-emphasis).
28. **"Export as CSV" / "Export as JSON" / "Create backup" / "Restore from backup" row labels** in Settings → Data section — these are `NavigationLink` / button labels in a system `Form` and use default `Color(.label)` text. The trailing chevron / system arrow is system gray. The accent does NOT appear on these rows because they are routine system list rows. The "Restore from backup" row is the only destructive row; its label is `Color.systemRed` to telegraph the destructive nature before the alert (matches iOS "Delete Account" Settings convention).
29. **"Go to session" link** in `ExercisePRsView` PR rows — when a PR row is tappable to jump to the session that achieved it, the row chevron (`chevron.right`) is `Color.accentColor` to match the affordance language of Phase 1 list rows. The row label itself stays `Color(.label)`.

**Explicitly NOT accent in Phase 6:**

- **All-set-average chart series** — uses `#3FBFC9` (secondary literal, NOT the project accent). Stroke width: 2pt solid for strength sessions, 1.5pt dashed (`StrokeStyle(lineWidth: 1.5, dash: [4, 3])`) for hypertrophy sessions. `PointMark` symbol size 40 (visually subordinate to top-set series).
- **`RuleMark` "previous best week" overlay** in `WeeklyTonnageView` — uses `Color(.secondaryLabel)` stroke (`StrokeStyle(lineWidth: 1, dash: [2, 4])`) and a `.caption` annotation in `Color(.secondaryLabel)`. The previous-best reference is informational, not a primary affordance.
- **Chart X-axis tick labels and Y-axis grid lines** — Swift Charts system default (gray). We do not customize.
- **Restore confirmation alert "Restore" destructive button** — uses system `.destructive` role (red), per iOS alert convention. Not accent.
- **PR banner dismiss-on-tap target** — no separate dismiss button or "X" glyph. The entire banner is the tap target; the dismiss affordance is communicated via the accessibility hint, not a glyph. Adding a glyph would imply the banner has multiple affordances; it has one (dismiss).
- **Empty-state action buttons** ("Define a block to see phase slicing", "Log 2 sessions to see your trend") — text-only, `Color(.secondaryLabel)`. These are explanatory copy, not actionable buttons (the user achieves the prompt by training; there is no in-view button to tap).
- **Δ value cell background in `SessionComparisonView`** — no fill; the Δ value is the only visual emphasis. Adding a background would compete with the row's natural list-row background.

**Layout decision (Claude's Discretion, D-37):** `ProgressHomeView` is a **single-column inset-grouped list with three sections** (Recent / Trends / Compare). Not a grid, not a hybrid card view. Rationale: (a) the per-exercise "card" data is shallow (name + 2 metadata fields) and adds no visual richness from a grid layout; (b) iOS conventions for "list of destinations from a tab root" overwhelmingly use inset-grouped list (Health, Files, Photos albums); (c) a 2-up grid at small font sizes would force exercise-name truncation; (d) inset-grouped naturally accommodates the three logical sections without custom section dividers.

## Interaction Patterns

### Progress tab navigation flow

1. User taps the "Progress" tab in `RootView.TabView` → `ProgressHomeView` is the root.
2. Each tab maintains its own `NavigationPath`. Tap-to-pop-to-root: when the user re-taps an already-active tab, that tab's `NavigationPath` is reset.
3. From `ProgressHomeView`:
   - Tap a per-exercise card → push `ExerciseProgressView` with that exercise.
   - Tap "Weekly Tonnage" → push `WeeklyTonnageView`.
   - Tap "Personal Records" → push `ExercisePRsView` with all-exercises scope.
   - Tap "This week vs last week" → push `SessionComparisonView` with the most recent completed session as anchor.
4. Deep links: `ExerciseDetailView` → "View progress" → push `ExerciseProgressView`. `ExerciseHistoryView` → toolbar `chart.line.uptrend.xyaxis` → push `ExerciseProgressView`. These deep links push onto the **originating tab's** NavigationPath, not the Progress tab's path (per per-tab `NavigationStack` rule).

### `ExerciseProgressView` series toggle flow

1. View loads → both `[Top set]` and `[All-set avg]` chips are ON by default (D-07).
2. Chart renders both series: top-set as solid line (strength) or dashed line (hypertrophy) per session; all-set-avg as a parallel pair of lines in `#3FBFC9`.
3. Tap a chip → chip toggles selected/unselected → `.selectionChanged` haptic (D-35) → the corresponding series is filtered out of the `Chart(filteredSeries)` array, triggering a re-render.
4. If both chips are OFF, an inline caption replaces the chart: "Select a series to display." (`.caption .secondaryLabel`, centered in the chart frame area).
5. Chip state is session-ephemeral (not persisted across navigation). Resetting the chip state requires returning to the view.

### Filter-chip row interaction (`WeeklyTonnageView`)

1. View loads → time range defaults to "Last 26 wk"; block phase chips all selected (when Phase 4 ships) or disabled (Phase 4 unshipped); muscle chips none selected (= single-color bars).
2. Tap a time-range chip → chip becomes selected (others in row deselect — single-select); `.selectionChanged` haptic; chart re-fetches via updated `Predicate<SetEntry>` date range.
3. Tap a block-phase chip → chip toggles selected/unselected (multi-select); `.selectionChanged` haptic; chart re-renders filtered by `BlockPhase`.
4. Tap a muscle chip → chip toggles selected/unselected (multi-select); `.selectionChanged` haptic; if `selectedMuscles.isEmpty` → single-color bar; otherwise → stacked bar via `.foregroundStyle(by: .value("Muscle", …))`.
5. Each filter row scrolls horizontally independently. The chart frame's vertical position is stable (rows don't wrap; height is constant 44pt per row).

### In-session PR banner flow (`InSessionPRBanner`)

1. Session starts → `SessionFactory.start(...)` seeds `InSessionPRState.prTable` with the current per-exercise per-intent PR table (one fetch + in-memory partition; per RESEARCH Pitfall 6).
2. User logs a working set → `SetEntry.isComplete` set to true → `PRDetector.check(set: justSavedSet, against: prState.prTable)` invoked → returns `Set<PRKind>` (possibly empty).
3. If non-empty: `prState.currentBanner = InSessionPRBannerContent(kinds: sortedCanonically, weight: set.weight, reps: set.reps)`. The `.safeAreaInset(edge: .top)` content view becomes visible (animation: slide-down + opacity, ≤200ms, or opacity-only when `accessibilityReduceMotion == true`).
4. `UINotificationFeedbackGenerator.notificationOccurred(.success)` fires on appearance.
5. PR table is mutated in-memory (the new set becomes the new PR for its bucket(s)) so subsequent set saves in the same session compare against the updated table.
6. Dismiss: tap banner → `prState.currentBanner = nil`. Or next set save → if next save also has PRs, replace banner contents; if next save has no PRs, banner dismisses. Or 5s timer → `prState.currentBanner = nil`.
7. Animation on dismiss: reverse of appearance.

### Restore flow (two-step destructive)

1. User taps Settings → Data → "Restore from backup" (row label is `.systemRed`).
2. `.fileImporter(allowedContentTypes: [.fitbodBackup])` opens. User selects a `.fitbodbackup` file from Files / iCloud Drive / AirDrop / on-device.
3. App starts security-scoped resource access. Pre-check 1: read manifest.json. If unreadable → "Couldn't read backup" alert.
4. Pre-check 2: verify SHA-256 of store.json matches manifest.checksum. If mismatch → "Backup is damaged" alert.
5. Pre-check 3: verify `manifest.schemaVersion == "v2"`. If mismatch → "Backup version not supported" alert.
6. Pre-check 4: verify no active `Session` in current container. If active → "Finish your session first" alert.
7. All pre-checks pass → present the destructive-confirmation alert: "Restore from backup? / This will replace all current data with the backup. This cannot be undone. / [Restore (destructive)] [Cancel]".
8. User taps "Cancel" → release security-scoped resource, no state change.
9. User taps "Restore" → "Restoring from backup…" modal sheet (non-cancellable). The restore coordinator:
   - Writes current container's full export to `documentDirectory/store-pre-restore-{ISO-timestamp}.json` (rollback insurance per D-32 step 4).
   - Deletes the current ModelContainer's SQLite store files.
   - Instantiates a fresh ModelContainer.
   - Decodes the backup's store.json into `ExportDocument` DTOs.
   - Inserts all entities, commits the model context.
   - Copies `images/` from the archive to the custom-exercise images directory.
10. Restore complete → dismiss the modal sheet → present the "Restore complete / Fitbod will restart to load your data. / [OK]" alert.
11. User taps "OK" → call `exit(0)` (per RESEARCH Pitfall 10; personal-app acceptable). App re-launches; SwiftUI re-mounts the entire view tree against the new ModelContainer.
12. Restore failure mid-step (rare) → restore coordinator catches the error → present "Restore didn't finish / Your previous data wasn't changed. The backup may be incomplete. / [OK]" alert. The side-file from step 9.1 means the user's data is still on disk; if the ModelContainer state is inconsistent, the next launch will detect it and offer a manual recovery (planner: defer this manual-recovery UI to a follow-up; for v1, the side-file is sufficient insurance).

## What's Explicitly Deferred (so the checker doesn't fail it for absence)

These are intentionally NOT in the Phase 6 UI contract — they belong to later phases or v2 per `ROADMAP.md` / `06-CONTEXT.md` / `<deferred>`:

- **Automated/scheduled backups** — v2.
- **CloudKit / iCloud sync of live data** — explicitly out-of-scope per `PROJECT.md`. Backup file is the only inter-device transfer mechanism.
- **Per-muscle PR records** ("best e1RM contribution to chest from a single set") — v2.
- **Velocity-based progress charts** — no VBT hardware in v1 per `PROJECT.md`.
- **Animated chart entry / scrubber overlay** — visual polish without information-density gain; deferred until usage shows a need.
- **Body-silhouette muscle heatmap** — Phase 5 owns; uses SwiftUI `Canvas`, not Swift Charts.
- **Weekly recap auto-surface** — Phase 5 owns; Phase 6 may link to it from `ProgressHomeView` if it exists.
- **Plateau-stall flag on exercise cards** — Phase 5 owns the signal (PROG-06); Phase 6 will read the signal if Phase 5 ships first, but does not compute it. The visual surface for a stall flag on a `ProgressHomeView` card is deferred to whichever phase ships last.
- **PR confetti / celebratory animation** — matches the project voice: direct, no exclamation points. No confetti.
- **Backwards-compatible schema migration on restore** — v1 restore requires exact schema-version match (`"v2"` only). When the schema starts versioning past v2 in production, a future phase adds a migration layer keyed off `manifest.schemaVersion`.
- **Custom export presets (date-ranged, exercise-filtered)** — v1 ships all-data exports only; filtered exports are v2.
- **Cross-platform `.fitbodbackup` interchange** — v1 backup is opened by Fitbod only. If a future phase needs Finder-unzippable backups, switch from AppleArchive `.aar` to raw ZIP.
- **PR banner queue / stack** — explicitly forbidden by D-16. Multiple PRs across consecutive saves replace, not stack.
- **`wasManualOverride` interaction with PR detection** — D-17 declares it doesn't suppress PR detection. The field doesn't exist in v1 (Phase 3 not shipped at Phase 6 plan-time per RESEARCH §Schema reality vs CONTEXT); so the contract is N/A and the banner fires regardless. When Phase 3 ships, no Phase 6 code change is needed — D-17's "doesn't suppress" rule is the no-op behavior.
- **Themable color schemes beyond the existing accent** — D-37 explicit.
- **Onboarding tutorial overlays** — D-37 explicit.
