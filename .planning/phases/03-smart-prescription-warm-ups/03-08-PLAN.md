ProgressionStrategy.prescribe signature (plan 03-05):
- (history: [HistoryPoint], targetRepsLow: Int, targetRepsHigh: Int, targetRPE: Double?, smallestIncrement: Double, plates: [(weight: Double, countPerSide: Int)], barWeight: Double, minCalibrationSets: Int, lastSessionRepsArray: [Int]? = nil, lastSessionWeight: Double?, lastSessionReps: Int?, lastSessionRPE: Double?, lastSessionDate: Date?) -> (weight: Double, explanation: PrescriptionExplanation)

The strategy is pure — SessionFactory must FETCH the history + lastSessionRepsArray + the lastSession scalars before calling. Plan reuses PreviousMatchingIntent for lastSession scalars; adds a new private helper `fetchHistoryPoints(exerciseID:intentRaw:context:) -> [HistoryPoint]` that fetches a bounded set (say 50 most recent isComplete + isWarmup==false sets for the exercise+intent pair) and back-calculates each e1RM via TuchschererTable.percent (skipping nil-RPE points per RESEARCH §Pitfall 7).

PlateInventory fetch: SessionFactory needs a `plateInventory(for: PlateEquipmentKind, context: ModelContext) -> PlateInventory?` helper. If missing, call PlateInventoryDefaults.make(for:unitSystem:) to construct a transient PlateInventory for the prescribe() call (NOT inserted into context — just used for rounding math).

To cache the PrescriptionExplanation per SessionExercise for the WhyThisWeightSheet downstream, SessionFactory needs somewhere to persist it. Options: (a) add a `prescriptionExplanationData: Data?` field to SessionExercise (additive — but plan 03-01 already shipped; adding now means a SchemaV4 OR amending V3 retroactively which is a smell), (b) recompute on demand in the view, (c) cache in an @Observable singleton service. Resolution: (b) — the SessionExerciseCard / WhyThisWeightSheet recompute the PrescriptionExplanation from a strategy call against the same inputs at view time. This is acceptable because the strategies are pure and fast; recomputation cost is < 1ms per exercise. Implementation: SessionExerciseCard owns a `private func currentExplanation() -> PrescriptionExplanation?` helper that re-invokes ProgressionStrategyFactory.make(for: se.progressionKind).prescribe(...) using fresh PreviousMatchingIntent + history fetches from the @Environment(\\.modelContext).

This recomputation policy means SessionFactory ONLY needs to set SessionExercise.prescribedWeight; everything else (the explanation displayed in WhyThisWeightSheet, the bumpOccurred flag, the calibrating status) is recomputed at render time. This avoids adding new fields to SchemaV3 and avoids cache-staleness bugs.

For BumpBanner triggering: SessionExerciseCard's currentExplanation() returns .bumpOccurred per the strategy. SessionExerciseCard owns `@State private var bannerDismissed: Bool = false`. BumpBanner is visible when currentExplanation?.bumpOccurred == true && !bannerDismissed. Tap-anywhere on the card sets bannerDismissed = true.

For SetRow + PrescriptionWeightCell + PlateStackDisclosure wiring:
- The existing SetRow weight TextField is replaced by PrescriptionWeightCell.
- The cell receives: $entry.weight (existing field — written through to actualWeight in the existing SetRow.onChange handler), entry.sessionExercise?.prescribedWeight (the prescribed value), the currentExplanation() from the parent card (passed down via a closure or computed @Binding), and $entry.wasManualOverride.
- PlateStackDisclosure is rendered as a sibling VStack child below the SetRow's main HStack, conditional on a `plateStackVisible: Bool` @State managed by the card or a shared expandedPlateSetID @Binding<UUID?>.
- The "tap weight cell area NOT info.circle" differentiation: PrescriptionWeightCell already wraps the info.circle Button so that area is excluded from the larger tap target. Surrounding HStack onTapGesture toggles plateStackVisible (only when NOT focused on the TextField).
