# Phase 3 — UI Design Contract

> Visual and interaction contract for Smart Prescription & Warm-ups. Generated by gsd-ui-researcher (autonomous mode).

This phase makes each working set's recommended weight **transparent and inspectable**. The primary new surfaces are: a prescribed-weight cell with an `info.circle` icon on every working `SetRow`, a `WhyThisWeightSheet` bottom-sheet showing the calculation breakdown, an inline plate-stack disclosure that unfolds beneath a tapped weight cell, a "You earned the weight bump" pill banner on the session exercise card header, a "calibrating" badge on RPE-autoreg exercises, a 4-set warm-up ramp that auto-inserts as warmup rows on the first qualifying compound, and a `PlateInventoryEditor` reachable from Settings.

The design stance is identical to Phases 1 and 2 verbatim: **"comprehensive but uncluttered"** — progressive disclosure is the tie-breaker when density and clutter conflict. Every new component stays on iOS-native chrome. No custom design system is introduced.

This UI-SPEC carries forward every token from Phases 1 and 2 (`01-UI-SPEC.md`, `02-UI-SPEC.md`) verbatim: accent `#0E7C86` / `#3FBFC9`, the 8-point spacing scale, the 4-role / 2-weight semantic typography, the 60/30/10 color split, and the terse-prescriptive copywriting voice. Anything below that does not restate a Phase 1/2 token is a Phase 3 addition.

| Property | Value |

**Why "none":** Unchanged from Phases 1 and 2. `PROJECT.md` locks the project to zero third-party SPM / UI dependencies. The shadcn registry model is React-only and not applicable to iOS native. No new frameworks are added in Phase 3 — the plate-stack visualization is implemented with SwiftUI `HStack` of styled `Rectangle` shapes (no `Canvas`, no UIKit; simple enough that Canvas is unnecessary overhead).

## Spacing Scale

**Carries forward verbatim from Phases 1 and 2.** Multiples of 4, iOS 8-point grid.

| Token | Value | Phase 3 usage |
|-------|-------|---------------|
| xs | 4pt | Gap between weight-cell value and `info.circle` icon; gap between individual plate rectangles in the stack visualization; "calibrating" dot-to-label gap |
| sm | 8pt | Vertical gap between prescription weight cell row and the plate-stack disclosure; internal padding of the bump banner pill; gap between warm-up set rows and the divider above working sets |
| md | 12pt | Vertical padding inside `WhyThisWeightSheet` content rows; warm-up row vertical padding (same as working `SetRow`); `PlateInventoryEditor` row vertical padding |
| lg | 16pt | Default horizontal inset in `WhyThisWeightSheet`, `PlateCalculatorSheet`, `PlateInventoryEditor`; primary content margin in Settings Smart Progression section |
| xl | 24pt | Section spacing inside `PlateInventoryEditor` between equipment tabs; spacing between the warm-up ramp block and the first working set in `SessionExerciseCard` |
| 2xl | 32pt | Empty-state vertical padding in `PlateInventoryEditor` when no plates are configured |
| 3xl | 48pt | (unchanged from Phase 2 — top spacer above empty-state heading on tall device) |

**Exceptions (load-bearing for Phase 3):**

- **44pt minimum touch target** carries forward for all tappable controls. Specifically: the `info.circle` icon button next to the prescribed weight cell must have a `.contentShape(Rectangle()).frame(minWidth: 44, minHeight: 44)` hit area even though the icon itself is ~20pt. The plate-stack disclosure tap target on the weight cell itself already satisfies 44pt (it inherits from `SetRow`'s weight `TextField` touch area). Each plate-inventory row's "delete plate" button must be 44pt. The "Reset to defaults" button in `PlateInventoryEditor` uses the system `.bordered` button style and is inherently ≥44pt.
- **Plate stack rectangle heights:** Each plate rectangle in the inline plate-stack visualization is 24pt tall and visually compact. The horizontal tap target for the disclosure is the entire weight-cell area (44pt high), not the plate rectangles themselves.
- **Bump banner height:** The "You earned the weight bump" pill banner is 44pt tall — intentionally touch-target height so it can be dismissed with a single tap.
- **Warm-up set rows** use the same 12pt vertical padding as working `SetRow`s. They are visually distinguished by the "W{N}" leading label and a `Color(.systemBlue)` set-type chip (matching the existing Phase 2 warm-up chip convention), NOT by a different height.

## Typography

**Carries forward verbatim from Phases 1 and 2.** Dynamic Type semantic styles, never hard-coded pt sizes. Exactly **4 roles**, exactly **2 weights** (regular / semibold). No bold weight.

| Role | SwiftUI semantic style | Approx pt @ Large | Weight | Line height | Phase 3 usage |
|------|------------------------|-------------------|--------|-------------|---------------|
| Caption | `.caption` | 12 | regular (400) | 1.3 | Plate weight labels under each rectangle in the plate stack; "calibrating (4 / 10 sets)" badge text; warm-up percentage hint (e.g. "40% × 5") shown faintly above each warm-up row in the set-row's Previous column area; "Why this weight?" sheet sub-labels (e.g. "Formula", "Rounded"); plate-inventory footer help text; bump banner dismiss hint |
| Body | `.body` | 17 | regular (400) | 1.4 | Prescribed-weight value in the weight cell; `WhyThisWeightSheet` content values (e.g. "100 kg × 8 @ RPE 8.5 (May 15)"); `PlateInventoryEditor` field values; plate-calculator target-weight display; calibration-range display (e.g. "95 – 105 kg") |
| Headline | `.headline` | 17 | semibold (600) | 1.3 | `WhyThisWeightSheet` row labels ("Last session", "Formula", "Computed", "Rounded", "Status"); `PlateInventoryEditor` section headers ("Bar weight", "Plates per side"); Settings section header "Smart Progression"; bump banner primary text ("Bumping to 102.5 kg") |
| Title2 | `.title2` | 22 | semibold (600) | 1.2 | (unchanged — empty-state headings only; no new Title2 surfaces added in Phase 3) |

No new typography role is added in Phase 3. The calibration range ("95 – 105 kg") uses `.body` regular — it is data, not a heading. The bump banner primary line uses `.headline` semibold to ensure readability as an inline dismissible notification.

### Accent reserved for (Phase 3 surfaces; appended to the Phase 1 + Phase 2 list)

The accent appears **only** at the following elements in Phase 3, in addition to all Phase 1 and 2 surfaces already listed. Every other interactive control uses default system styling.

Phase 1 surfaces (still reserved): items 1–7 from `01-UI-SPEC.md`.
Phase 2 surfaces (still reserved): items 8–17 from `02-UI-SPEC.md`.

Phase 3 additions:

18. **`info.circle` icon** next to the prescribed-weight value cell on each working `SetRow` — accent foreground, never `.secondaryLabel`. This is a primary disclosure affordance, not an incidental glyph. Tapping it opens `WhyThisWeightSheet`.
19. **"Calibrated" status badge background** in `WhyThisWeightSheet` — a small capsule with `Color.accentColor.opacity(0.15)` fill and accent-colored label text ("calibrated"), matching the Phase 1 "Custom" tag treatment. The "calibrating" badge (pre-threshold) uses `Color(.systemGray5)` fill with `.secondaryLabel` text — it is NOT accent because it is a transitional / incomplete state.
20. **"Apply Prescription" or "Use Suggested Weight" fill button** inside `WhyThisWeightSheet` if the user has already diverged from the prescription and wants to revert — accent background, white label. Only present when `SetEntry.wasManualOverride == true` for this set's entry. Absent otherwise (the sheet is read-only when the prescription is already applied).
21. **"Reset to defaults" button** in `PlateInventoryEditor` per equipment tab — text-only, accent foreground (matching the Phase 1 "Copy as Custom" / Phase 2 "+ Folder" text-button treatment). No fill. Tapping restores the default plate list for that equipment kind.

**Explicitly NOT accent in Phase 3:**
- The bump banner ("You earned the weight bump") background — uses `Color(.systemGreen).opacity(0.15)` fill with `Color(.label)` text. It is a success/affirmation state, not a primary action. Using accent here would conflict with the "accent = user-initiated action" contract.
- Plate rectangles in the stack visualization — use semantic system colors (see Asset Contract below), never the app accent. The visualization is informational.
- The "calibrating" badge — `Color(.systemGray5)` fill, `.secondaryLabel` text. Transitional state.
- The "warm-up" set-type chip on warm-up rows — stays `Color(.systemBlue)` per the Phase 2 set-type chip color convention ("warmup = `.systemBlue`"). Phase 3 does not alter the chip color contract.
- The `info.circle` icon on calibrating exercises — same accent treatment as calibrated exercises; the badge indicates the calibration state, the icon indicates the disclosure affordance. Both are separate elements.
- The "just this session" toggle in `WarmupConfigSheet` — system `Toggle` tint (follows accent automatically via `.tint`). Not explicitly called out as a Phase 3 accent addition because it is covered by Phase 1's "Active Toggle in Settings" rule.

## Copywriting Contract

All copy is **terse and prescriptive**, matching the Phase 1 and 2 voice: second-person, direct, no exclamation marks, no marketing register. The strings below are normative — the executor copies them verbatim from this file into SwiftUI source. Where a value is runtime-computed (e.g. weight, reps), the format string is shown with `{placeholder}` syntax.

### Prescribed weight cell and `info.circle` icon

The prescribed-weight cell on each working `SetRow` shows the rounded prescription from the progression strategy. It doubles as the numeric weight input (user can type to override). The `info.circle` icon appears inline to the right of the value.

| Element | Copy |
|---------|------|
| Weight cell — prescription shown, unmodified | (the numeric value, e.g. "102.5 kg" — no label; the value IS the affordance) |
| Weight cell — manual override applied (user has typed a different value) | (the user-typed numeric value; an "M" badge in `.caption .secondaryLabel` appears to the right of the cell, left of the `info.circle` icon, to signal override) |
| `info.circle` button accessibility label | "Why this weight?" |
| `info.circle` button accessibility hint | "Shows how this weight was calculated." |

### Destructive actions in Phase 3

Phase 3 introduces two destructive surfaces:

| Action | Surface | Confirmation approach |
|--------|---------|----------------------|
| Delete a plate from inventory | `PlateInventoryEditor` row swipe-to-delete | System swipe-to-delete (no extra confirmation — low blast radius; user can re-add); "Delete" label in system red |

## Interaction Patterns

### "Why this weight?" disclosure flow

1. User enters `SessionLoggerView`. Each working `SetRow` (not warm-up rows) shows the prescribed weight in the weight cell plus an `info.circle` icon.
2. User taps the `info.circle` → `WhyThisWeightSheet` slides up to `.medium` detent. The sheet title is "Why this weight?" Navigation back is swipe-down.
3. Sheet content is a `List(.insetGrouped)` with the data rows. If `PrescriptionExplanation.status == .calibrating(n, threshold)`, the status row shows "Calibrating ({n} / {threshold} sets)" in `.body .secondaryLabel`. If `.calibrated`, the row shows "Calibrated" on a `Color.accentColor.opacity(0.15)` capsule.
4. If `wasManualOverride == true` for the current set, a "Use Suggested Weight" accent-filled button appears as a `Section` footer at the bottom of the sheet. Tapping it writes the `roundedWeight` back to `SetEntry.actualWeight`, sets `wasManualOverride = false`, and dismisses the sheet.
5. If `wasManualOverride == false`, no button appears; the sheet is fully read-only.
6. Tapping outside the sheet (iOS sheet dismiss gesture) collapses it without any state change.

### Plate-stack inline disclosure flow

1. User taps the weight cell area (not the `info.circle` icon specifically) of a `SetRow`. The `info.circle` tap is differentiated from the weight-cell-area tap via `.simultaneousGesture` + hit testing on the icon subview.
2. `PlateStackDisclosure` slides in below that row with a 200ms ease-in-out height animation. The plate stack shows the current weight value (from `SetEntry.actualWeight` if the user has entered a value, otherwise the prescribed weight from `SetEntry.prescribedWeight`).
3. The disclosure shows the bar center segment, then plates per side in descending weight order. If no solution exists, the "no combination found" error caption appears instead.
4. Tapping the weight cell area again collapses the disclosure. Tapping a different row's weight cell opens that row's disclosure and collapses this one (mediated by the `expandedPlateSetID` binding on the parent `SessionExerciseCard`).
5. The disclosure does NOT interfere with text entry in the weight cell — if the user taps directly on the text cursor area (SwiftUI `TextField` focus), the `TextField` gets focus and the plate stack stays open (or opens if it was closed). The disclosure collapses only when a different row's cell is tapped or the same cell is tapped while the keyboard is not active.

### Bump banner flow

1. At session start, `SessionFactory.start(...)` invokes `ProgressionStrategy.prescribe(...)` per `SessionExercise`. If `DoubleProgressionStrategy` detects the bump condition (all working sets from last session hit the top of the rep range), it sets a `bumpOccurred: Bool = true` flag in the returned `PrescriptionExplanation`.
2. `SessionExerciseCard` renders `BumpBanner` at its top when `bumpOccurred == true` and `bannerDismissed == false` (local `@State`).
3. The banner is dismissed (set `bannerDismissed = true`) on the user's first tap anywhere on the exercise card — including any `SetRow` tap, weight cell tap, or reps cell tap within that card. The `onTapGesture` on the card's outer container handles this.
4. The banner does NOT re-appear in the same session after dismissal. It does NOT appear for calibrating RPE-autoreg exercises (only for double-progression).

### Warm-up ramp insertion flow

1. `SessionFactory.start(...)` calls `WarmupRamp.shouldGenerate(for:in:)` for each `SessionExercise` in order. The first qualifying `SessionExercise` (mechanic == compound AND equipment ∈ {barbell, dumbbell} AND working weight ≥ 1.5× bar weight AND not bodyweight AND deload flag is false AND `WarmupConfig.enabled != false`) gets ramp generation.
2. `WarmupRamp.generate(top:bar:plates:)` returns 4 `SetEntry` rows (2 for dumbbells) with `isWarmup = true`, `setTypeRaw = "warmup"`, `weight` = plate-rounded ramp percentages, `reps` = per-ramp-step rep target (5/3/2/1), `isComplete = false`.
3. These `SetEntry` rows are inserted into the `SessionExercise.setEntries` array BEFORE the planned working-set rows, with `orderIndex` 0, 1, 2, 3 (warm-ups) followed by 4, 5, 6… (working sets).
4. `SessionExerciseCard` reads `setEntries` sorted by `orderIndex`; rows with `isWarmup == true` render via `WarmupRampRows`, rows with `isWarmup == false` render via the existing working-set loop.
5. Tapping "Skip warm-ups" on the `WarmupRampRows` component: sets all `SetEntry.isComplete = true` and `SetEntry.weight = 0` on the warm-up rows (marks them done without weight — they remain in history but with `isWarmup = true` and `isComplete = true` so they don't affect progression), and collapses the warm-up block to a single strikethrough caption "Warm-ups skipped."
6. If `WarmupConfig.skipNextSession == true` at session start, `WarmupRamp.shouldGenerate(...)` returns false for this exercise, no warm-up rows are inserted, and `WarmupConfig.skipNextSession` is reset to false immediately after. The "warm up however you'd like" placeholder row is shown in its place (via a single read-only row tagged with a recognizable sentinel value, not an actual `SetEntry`).

### `PlateInventoryEditor` navigation

- Reachable from `SettingsView` → "Smart Progression" section → "Plate Inventory" row → push `PlateInventoryEditor` onto the Settings `NavigationStack`.
- Tabbed layout: SwiftUI `TabView` with `tabViewStyle(.segmented)` for the four equipment kinds. Each tab is a `Form(.insetGrouped)`.
- "Add Plate" action: presents a `.sheet` with a single-field form (weight field + "Add" button). The new `PlateSpec` is appended to the current tab's `PlateInventory.availablePlates` array sorted descending by weight.
- Swipe-to-delete on a plate row: system swipe-to-delete, no extra confirmation (low blast radius).
- "Reset to defaults" button at the bottom of the form (section footer or trailing toolbar button, TBD by planner based on SwiftUI `Form` idioms): presents an `.alert` with destructive confirmation before overwriting.
- All edits write through to the `PlateInventory` SwiftData entity immediately (write-through binding, same `@Bindable` pattern as the rest of the app). No "Save" button — edits are live.
- The "Try it" `PlateCalculatorSheet` is accessible via a toolbar trailing button ("Calculator") on the navigation bar of `PlateInventoryEditor`.

## Asset Contract

### Color palette — plate rectangles

The plate-stack visualization uses a fixed semantic color scheme to distinguish plate weights at a glance. These use SwiftUI system colors (no new asset catalog entries).

| Plate weight tier | Color token | SwiftUI value |
|-------------------|-------------|---------------|
| ≥ 25 kg (or ≥ 45 lb) | Red | `Color(.systemRed).opacity(0.8)` |
| 20 kg (or 35 lb) | Blue | `Color(.systemBlue).opacity(0.8)` |
| 15 kg (or 25 lb) | Yellow | `Color(.systemYellow).opacity(0.8)` |
| 10 kg (or 25 lb dumbbell handle plates) | Green | `Color(.systemGreen).opacity(0.8)` |
| 5 kg (or 10 lb) | White / Light | `Color(.systemGray5)` |
| 2.5 kg (or 5 lb) | Gray | `Color(.systemGray3)` |
| < 2.5 kg (microplates) | Silver | `Color(.systemGray6)` with a `Color(.systemGray2)` border |
| Bar segment (center) | Chrome/Gray | `Color(.systemGray4)` |

This color scheme matches the unofficial-but-widely-recognized international weight plate color convention (IWF-adjacent). A lifter will parse it instantly. The `.opacity(0.8)` prevents the vivid colors from overpowering the system background in both light and dark mode.

The badge is decorative from the card's perspective — mark it `.accessibilityHidden(false)` and `.accessibilityAddTraits(.isStaticText)` so VoiceOver reads it in the card element ordering. |

## What's Explicitly Deferred (so the checker doesn't fail it for absence)

These are intentionally NOT in the Phase 3 UI contract — they belong to later phases per `ROADMAP.md` / `03-CONTEXT.md` / `<deferred>`:

- **Block-periodized and Hybrid progression strategies** (Phase 4) — the "Block Periodized" and "Hybrid" progression picker entries remain in the routine builder (for forward compatibility) but still silently behave like "Double Progression" in Phase 3 sessions. The Phase 2 footnote `"Block / Hybrid models activate when Block Periodization ships in Phase 4."` remains in `.caption .secondaryLabel` under the picker rows.
- **Deload-week warm-up suppression** (Phase 4) — `WarmupRamp.shouldGenerate(...)` has the deload-flag conditional stub but the flag is always `false` in Phase 3 (no block entity is live yet). When Phase 4 ships the deload signal, only the flag's source changes; the UI path is already wired.
- **Plateau detection flag on exercise cards** (Phase 5) — no stall badge in Phase 3.
- **"M" badge in history rows** (Phase 6 polish) — the `wasManualOverride` flag is written in Phase 3 but no history-row badge surfaces until Phase 6.
- **Velocity-based loading** — explicitly out of scope per `PROJECT.md`.
- **Smart-rounding heuristics** (e.g. "round to 5 kg for high-rep sets") — deferred until usage shows a need.
- **`PlateCalculatorSheet` as a standalone reachable from anywhere** (e.g. a dedicated Settings row or a toolbar button in the session logger outside of the inventory editor) — the sheet exists as a "Try it" affordance in `PlateInventoryEditor` but is not a top-level navigation destination in Phase 3. Promoting it is deferred.
- **Custom progression strategies** — out of v1 entirely.
- **Per-exercise time-series charts** (Phase 6) — `ExerciseHistoryView` remains list-only.
