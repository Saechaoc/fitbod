<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User taps Delete → cascade write | Deleting a block via swipe-action triggers a cascade delete of BlockPhase rows and nullifies Routine.block / Session.block via SwiftData's inverse defaults. The confirmation alert is the explicit consent gate. |
| Template selection → BlockDraft seed | User picks a template from the Menu; the corresponding `BlockTemplate` value type is passed to `BlockDraft(template:)`. Templates are static (no user-mutable inputs at selection time). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Tampering | Delete confirmation alert can be bypassed if user taps the swipe-delete chrome instead of the confirm Button | accept | SwiftUI's swipe-to-delete idiom is the platform convention; the explicit `.alert` adds a second confirmation step. Standard iOS UX. |
| T-04-03-02 | Information Disclosure | The delete confirmation message references "linked routines stay" — implies some routines exist; not sensitive data | accept | Single-user app; no multi-tenant disclosure. |
| T-04-03-03 | Denial of Service | Deleting a Block with many sessions cascades through inverse nullify on Session.block — large block could pause UI briefly | mitigate | SwiftData's inverse-nullify on `.cascade` parent delete is implemented as a bulk update; for realistic block sizes (≤100 sessions over an 8-week meso) the operation is < 100ms. No further mitigation required. |
| T-04-03-04 | Repudiation | Deletion has no undo | accept | Standard iOS pattern; user-confirmed via alert. |
| T-04-03-05 | Elevation of Privilege | None — section is read/write owned by the same user who owns the rest of the app | accept | Single-user local app. |
</threat_model>
