<objective>
Wire the Routine ↔ Block linkage so the user can assign a routine to a block from the RoutineBuilderView header, and so the per-exercise progression Picker conditionally exposes `.block` / `.hybrid` only when the routine is in a block (D-19). After this plan, a routine inside a block has all four progression strategies available; a routine without a block sees only `.rpe` and `.double` and an inline footnote prompts the user to add the routine to a block if they want the block-aware kinds.

Purpose: Plan 04-06 ships BlockPeriodizedStrategy + HybridStrategy that consume `Session.block` (snapshotted in plan 04-07). For those strategies to be selectable per RoutineExercise, the user needs an entry point to set `routine.block` — and the picker needs to expose the right options. This plan delivers both surfaces.

Output: One new BlockPickerMenu view, three modified Phase 2 files (RoutineDraft / RoutineBuilderView / PrescriptionEditorRow), one modified Phase 2 RoutineExerciseCard (propagates hasBlock), two new test suites.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User picks Block from Menu → RoutineDraft.blockID write | Local user input, no untrusted source. The fetch-by-id on save normalizes the binding to a real Block or nil. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-05-04 | Spoofing | RoutineDraft's blockID write does not require explicit confirmation | accept | The picker IS the explicit confirmation. No additional gate needed. |
