# Phase 4 — UI Design Contract

> Visual and interaction contract for Periodization & Blocks. Generated by gsd-ui-researcher (autonomous mode).

This phase delivers **defined mesocycles with deterministic phase progression and a transparent home-screen timeline**. The primary new surfaces are: a `BlockBuilderView` mirroring `RoutineBuilderView` (single-screen template editor with inline phase rows), a "Blocks" section on `RoutinesListView` above existing folders, a `BlockCard` rendered above `ResumeWorkoutBanner` on the Today tab when a block is active (with horizontal-swipe `TabView` paging between mesocycle weeks), a top-of-Today `Deload week` banner during deload weeks, a `BlockReviewView` sheet that surfaces at end-of-block, a `ConsiderDeloadBanner` scaffolded for the Phase 5 fatigue signal, and a "Block" picker in the `RoutineBuilderView` header. The Phase 3 `WhyThisWeightSheet` gains two new strategy explanation variants (Block Periodized + Hybrid) — same component, new sentences.

The design stance is identical to Phases 1, 2, and 3 verbatim: **"comprehensive but uncluttered"** — progressive disclosure is the tie-breaker when density and clutter conflict. Every new component stays on iOS-native chrome. No custom design system is introduced.

This UI-SPEC carries forward every token from Phases 1, 2, and 3 (`01-UI-SPEC.md`, `02-UI-SPEC.md`, `03-UI-SPEC.md`) verbatim: accent `#0E7C86` / `#3FBFC9`, the 8-point spacing scale, the 4-role / 2-weight semantic typography (`.caption` / `.body` / `.headline` / `.title2`), the 60/30/10 color split, and the terse-prescriptive copywriting voice. Anything below that does not restate a Phase 1/2/3 token is a Phase 4 addition.

## Spacing Scale

**Carries forward verbatim from Phases 1–3.** Multiples of 4, iOS 8-point grid.

| Token | Value | Phase 4 usage |
|-------|-------|---------------|
| xs | 4pt | Gap between phase-chip dot and label inside `BlockCard`; gap between block name and "Week N of M" badge in `RoutinesListView` block rows; gap between multipliers and their suffix labels ("×0.85 vol", "×0.88 int") on the block card week page |
| sm | 8pt | Horizontal gap between phase chips inside the `BlockPhaseEditorRow`; vertical gap between block-name field and start-date field in `BlockBuilderView`; vertical gap between adjacent `BlockPhaseEditorRow`s; gap between `BlockCard` content blocks (phase chip → week badge → days-remaining → multipliers preview → scheduled-routines list) |
| md | 12pt | Internal padding of `BlockCard` (top/leading/trailing/bottom of the inner content area inside the card surface); vertical padding inside `BlockReviewView` content rows; vertical padding of each `BlockPhaseEditorRow` (matches Phase 2 `PrescriptionEditorRow` height); deload-banner internal vertical padding |
| lg | 16pt | Default horizontal inset of `BlockBuilderView`, `BlockReviewView`, `BlockCard` outer margin from screen edge; primary content margin in the deload banner; horizontal inset of the `ConsiderDeloadBanner` |
| xl | 24pt | Vertical spacing between `BlockReviewView` sections (Total Volume / e1RM Deltas / PRs Hit / Recommended Next Phase); spacing between the `BlockCard` and the `ResumeWorkoutBanner` below it on Today |
| 2xl | 32pt | Empty-state vertical padding inside `RoutinesListView` "Blocks" section when no blocks exist; vertical padding above the start-block CTA on Today when no block is active |
| 3xl | 48pt | (unchanged from Phase 3 — top spacer above empty-state heading on tall device) |

**Exceptions (load-bearing for Phase 4):**

- **44pt minimum touch target** carries forward for all tappable controls. Specifically: the phase-chip kind picker (`Menu`) inside `BlockPhaseEditorRow` must have a `.contentShape(Rectangle()).frame(minHeight: 44)` hit area even though the chip itself reads as ~32pt of visible height; the block-card swipe pager's left/right edges must accept a horizontal drag without conflicting with the underlying scroll view; the "Start [Recommended] Block" CTA in `BlockReviewView` uses the system `.borderedProminent` button style and is inherently ≥44pt; the deload-banner dismiss-chevron tap target (if present per D-13 — note that the deload banner is **non-dismissible** for visibility; see Copywriting Contract) is N/A in Phase 4.
- **BlockCard height:** `BlockCard` content occupies a single screen-width-spanning card with a fixed minimum height of ~140pt to ensure the phase chip, week badge, days-remaining row, and multipliers row all fit without truncation at Dynamic Type Large. At larger Dynamic Type sizes, the card grows vertically (no fixed-height truncation).
- **Mesocycle pager page height:** Each `TabView` page inside the `BlockCard` matches the card's `minHeight`; the page indicator dots are disabled (`indexDisplayMode: .never`), so no extra bottom padding is required.
- **Phase chip dimensions:** The phase chip on `BlockCard` and `BlockPhaseEditorRow` reuses the existing Phase 2 `IntentFilterChipRow` chip styling — `Text` with `.font(.caption)`, `.padding(.horizontal, 12)`, `.padding(.vertical, 6)`, `.clipShape(Capsule())`, and a `.frame(minWidth: 44, minHeight: 44)` hit area. Different content (`BlockPhaseKind` instead of `Intent`) but identical shape so the visual language stays coherent across Phase 2 and Phase 4.

### Phase color palette (D-07, locked by this UI-SPEC)

The four `BlockPhaseKind` cases render with the following colors. These appear on: the phase chip (capsule fill when "current phase"), the block-card background tint (subtle, ~15% opacity), the mesocycle week-strip per-week tint inside the swipe pager, and the deload banner background tint.

| Phase | Chip fill (current week) | Card background tint (~15% opacity) | Token / source |
|-------|--------------------------|--------------------------------------|----------------|
| Accumulation | `Color.accentColor` (= `#0E7C86` / `#3FBFC9`) | `Color.accentColor.opacity(0.15)` | App accent (`AccentColor`) — reused so the brand thread stays visible in the most common state |
| Intensification | `Color(.systemOrange)` adjusted to `#F59E0B` | `Color(red: 0.96, green: 0.62, blue: 0.04, opacity: 0.15)` | System amber-family (D-07; matches iOS system-amber visual weight) |
| Realization | `Color(.systemOrange)` adjusted to `#EA580C` | `Color(red: 0.92, green: 0.34, blue: 0.05, opacity: 0.15)` | System orange-family (D-07) |
| Deload | Custom hex `#94A3B8` (desaturated slate-gray) | `Color(red: 0.58, green: 0.64, blue: 0.72, opacity: 0.15)` | Desaturated gray — reads as "rest" (D-07) |

**Why these specific hex values:** Accumulation reuses the existing app accent so the most common training state stays visually aligned with the app's brand. Intensification (`#F59E0B`) and Realization (`#EA580C`) escalate through the warm-color family (amber → orange) to communicate increasing intensity. Deload (`#94A3B8`) deliberately desaturates to communicate "rest, don't load" — the opposite signal from the accent/warm spectrum. All four values pass WCAG AA contrast against `.systemBackground` in both light and dark mode at the `.caption` size used for chip labels (verified against the `#0E7C86` ↔ `#3FBFC9` adaptive pair already in use since Phase 1).

**Asset implementation note:** These four phase colors are NOT added to `Assets.xcassets`. They live as a static `enum BlockPhaseColors` in `fitbod/Periodization/BlockPhaseColors.swift` returning `Color` values inline. This keeps the asset catalog free of phase-specific entries and lets the executor evolve the palette without an asset-catalog round trip. Dark-mode tweaks are handled by `.opacity` adjustments (same pattern as Phase 3's plate-rectangle color scheme).

### Accent reserved for (Phase 4 surfaces; appended to the Phase 1 + 2 + 3 list)

The accent appears **only** at the following elements in Phase 4, in addition to all Phase 1, 2, and 3 surfaces already listed. Every other interactive control uses default system styling.

Phase 1 surfaces (still reserved): items 1–7 from `01-UI-SPEC.md`.
Phase 2 surfaces (still reserved): items 8–17 from `02-UI-SPEC.md`.
Phase 3 surfaces (still reserved): items 18–21 from `03-UI-SPEC.md`.

Phase 4 additions:

22. **Phase chip fill — only when the chip represents the CURRENTLY ACTIVE phase of the active block AND that phase is `.accumulation`.** Accumulation chips use the app accent because accumulation is "you are here, building." Other phases use their own non-accent palette colors (intensification = `#F59E0B`, realization = `#EA580C`, deload = `#94A3B8`). The chip in the `BlockBuilderView` editor row (`BlockPhaseEditorRow`) is never accent — those represent template configuration, not "you are here."
23. **"Save" button** in `BlockBuilderView` toolbar — accent foreground when the draft is valid (matches `RoutineBuilderView` "Save" Phase 2 convention). Greyed (`.disabled` system styling) when validation fails.
24. **"Start [Recommended] Block" CTA** inside `BlockReviewView` — accent background, white label (matches the Phase 1 "Save Custom Exercise" / Phase 2 "Start Workout" treatment). Tapping seeds `BlockBuilderView` with the stock template for the recommended next phase chain.
25. **"+ Block" menu trigger** in the "Blocks" section header on `RoutinesListView` — accent foreground text button (matches the Phase 2 "+ Folder" treatment in the routines header).
26. **"Add to Block" / "Block: {name}" picker** in `RoutineBuilderView` header — text-only, accent foreground when a block IS assigned (signals "this routine is in a block"); `.secondaryLabel` when none is assigned (signals "this routine has no block context"). Matches the Phase 2 "Folder: {name}" picker treatment.

**Explicitly NOT accent in Phase 4:**

- **Phase chips in `BlockPhaseEditorRow` (template editor):** Use the phase's own palette color (accumulation = accent, intensification = `#F59E0B`, realization = `#EA580C`, deload = `#94A3B8`) at full opacity. The "active" treatment doesn't apply because no block in the editor is yet running.
- **Phase chips in past weeks of the swipe pager:** Use the phase color at full opacity (NOT desaturated). Past weeks render the phase color as a visual record of what was scheduled.
- **Phase chips in future weeks of the swipe pager:** Same — phase color at full opacity. The "current week" signal is the linear `Week N of M` badge highlight + the card background tint, NOT a chip color change.
- **Deload banner background:** Uses the deload phase color at `opacity(0.15)` — `#94A3B8.opacity(0.15)`, NOT accent. The banner is informational/cautionary, not an action.
- **`ConsiderDeloadBanner` (advisory):** Uses `Color(.systemYellow).opacity(0.15)` background — distinct from the scheduled-deload banner so the user can tell at a glance which is which. Yellow connotes "consider," not "commanded." NEVER accent — this is an advisory the user must explicitly accept or dismiss.
- **`BlockReviewView` placeholder copy for the PRs section** ("Coming in Phase 6") — uses `.secondaryLabel`, never accent. Placeholders are not affordances.
- **Multipliers preview text** ("×1.0 vol · ×0.75 int") on `BlockCard` — uses `.secondaryLabel`, never accent. It's reference data, not an action.
- **Mesocycle pager dots:** Disabled entirely (`indexDisplayMode: .never`). The linear week badge inside each page is the canonical "which week" signal; an extra dot row would be redundant.
- **`Week N of M` badge text:** Uses default `.primary` foreground inside a `Color(.systemGray6)` capsule. NOT accent — the **chip** carries the "this is the current period" signal, the badge carries the numeric context.

## Copywriting Contract

All copy is **terse and prescriptive**, matching the Phase 1, 2, and 3 voice: second-person, direct, no exclamation marks, no marketing register. The strings below are normative — the executor copies them verbatim from this file into SwiftUI source. Where a value is runtime-computed (e.g. week index, multipliers), the format string is shown with `{placeholder}` syntax.

### `BlockCard` (Today tab — above `ResumeWorkoutBanner` when a block is active)

The `BlockCard` is a grouped-inset card on the Today tab that surfaces above `ResumeWorkoutBanner`. It contains a horizontal-swipe `TabView` whose pages are one-per-week of the entire mesocycle. The CURRENT week page is the initial selection; the user can swipe to past or future weeks for reference.

| Element | Copy |
|---------|------|
| Card header — block name (left-aligned headline) | "{block.name}" (e.g. "Powerlifting Peak Q3") |
| Card header — "End block" overflow menu (trailing) | "End Block" / "Edit Block" (`Menu` with two entries; "End Block" is destructive) |
| Per-week page — phase chip label | "Accumulation" / "Intensification" / "Realization" / "Deload" (matches the `BlockPhaseKind` raw values, title-cased) |
| Per-week page — linear week badge | "Week {N} of {M}" — e.g. "Week 5 of 8" |
| Per-week page — days-remaining caption (current week only) | "{N} days remaining" — e.g. "12 days remaining". For past weeks: "Completed". For future weeks: "Starts in {N} days". |
| Per-week page — multipliers preview caption | "×{volume} vol · ×{intensity} int" — e.g. "×1.0 vol · ×0.75 int" |
| Per-week page — scheduled routines list section header | "Routines this week" |
| Per-week page — scheduled routines list row | "{routine.name}" (with disclosure chevron — tap navigates into the routine builder) |
| Per-week page — no routines scheduled | "No routines linked to this block." (`.caption .secondaryLabel`) |
| Card accessibility label (root) | "Active block: {block.name}, currently week {N} of {M}, {phaseKind} phase, {N} days remaining." |
| Card accessibility hint | "Swipe horizontally to navigate between weeks of the mesocycle." |

## Interaction Patterns

### `BlockCard` swipe pager flow

1. User opens the Today tab. `BlockCard` appears at the top (above `ResumeWorkoutBanner`) when `@Query<Block>(filter: #Predicate { $0.isActive })` returns a row.
2. The card's internal `TabView(selection: $selectedWeekIndex)` is initialized with `selectedWeekIndex = PeriodizationEngine.weekIndex(for: activeBlock, on: Date.now) ?? 0`. The user lands on the current week.
3. User swipes left → moves to next week (e.g. Week 5 → Week 6). User swipes right → moves to previous week. The phase chip color updates per the destination week's phase. The card background tint also updates to the destination week's phase tint at 15% opacity.
4. Each `MesocycleWeekPage` shows: phase chip → `Week N of M` badge → days-remaining caption (only for the current week page) → multipliers preview → scheduled-routines list. Tapping a routine row navigates into `RoutineBuilderView` for that routine.
5. The `indexDisplayMode: .never` setting hides the default page dots. The linear week badge inside each page is the canonical "which week" indicator.
6. Card accessibility label is computed for the current week: "Active block: {name}, currently week {N} of {M}, {phaseKind} phase, {N} days remaining." VoiceOver users use the accessibility rotor to navigate between week pages.

### `BlockBuilderView` flow

1. User opens via: `+ Block` menu in `RoutinesListView` "Blocks" section header → menu picks a template (Generic / Hypertrophy / Powerlifting Peak / Blank) → `BlockBuilderView` opens modally pre-seeded with the picked template.
2. Or: user taps "Edit Block" in the `BlockCard` overflow menu → `BlockBuilderView` opens in edit mode bound to the existing `Block`.
3. The view layout mirrors `RoutineBuilderView`: name field + start-date + active-toggle at the top, then a `Section("Phases")` with one `BlockPhaseEditorRow` per `BlockPhaseDraft` in the draft's `phases` array.
4. Inside each `BlockPhaseEditorRow`: phase-kind chip (`Menu`-backed; tapping opens a 4-option menu to change kind), weeks `Stepper` (1–12), volume-multiplier `Stepper` (0.3–1.5 step 0.05), intensity-multiplier `Stepper` (0.3–1.1 step 0.01). The drag handle on the leading edge enables `.onMove` reorder via `EditMode + .constant(.active)` (same pattern as `RoutineBuilderView` Phase 2). Swipe-to-delete on the row removes the phase from the draft.
5. The "+ Phase" button at the bottom of the phases section adds a new `BlockPhaseDraft` to the array with the picked kind and that kind's RP-default multipliers (D-10 values applied automatically).
6. The "Total: {N} weeks across {M} phases" footer updates live as the user edits weeks values.
7. Toolbar trailing "Save": runs validation (empty name → caption error, no phases → caption error), then calls `BlockDraft.save(into: modelContext)`. The save wraps `try modelContext.transaction { ... }`: if `isActive == true`, first set every other `Block.isActive = false` (single-active invariant D-05); then materialize `Block` + `BlockPhase` rows from the draft; then `try modelContext.save()`.
8. On save success the view dismisses; on save failure the inline `.systemRed` "Couldn't save block. Try again." footer appears below the toolbar.
9. Toolbar leading "Cancel": prompts no confirmation (drafts are ephemeral and recoverable by re-picking the template); dismisses immediately.

### `BlockReviewView` trigger and acknowledgment flow

1. User opens the Today tab. On `TodayView.task`, the view checks `@Query<Block>` for any block where `endDate < Date.now AND isActive == true AND reviewedAt == nil`. Exactly one expected (other constraints prevent multiple).
2. If found, `TodayView` presents `BlockReviewView(block: matchedBlock)` as a `.sheet(isPresented: $reviewing)` with `.large` detent.
3. The sheet renders four sections in order: Total volume (computed inline from `SetEntry` `@Query` filtered to `block.startDate ... block.endDate`), e1RM deltas (start-of-block vs end-of-block top working set per exercise via Phase 3's e1RM helper), PRs hit (placeholder copy), Recommended next phase (via `PeriodizationEngine.recommendedNextKind`).
4. Tapping "Done" in the sheet toolbar: in a single transaction, set `block.reviewedAt = Date.now` AND `block.isActive = false`, then `try modelContext.save()`. The sheet dismisses. The block subsequently appears in `RoutinesListView` "Blocks" section with a "Started {date} · Ended {date}" caption.
5. Tapping the "Start [Recommended] Block" CTA: dismisses `BlockReviewView`, presents `BlockBuilderView` modally seeded with the stock template for the recommended kind chain. The user can then customize and save. Importantly, this does NOT auto-activate the new block — the user must explicitly toggle "Active block" in the builder.
6. Subsequent opens of the Today tab do NOT re-surface `BlockReviewView` for the same block (the `reviewedAt != nil` predicate excludes it).

### `RoutineBuilderView` Block picker flow

1. User opens `RoutineBuilderView` (create or edit). A new "Block:" row appears in the header section beneath the folder picker.
2. Tapping the row opens a `Menu` listing: "No Block" + all defined blocks (active block listed first with a leading accent dot) + "+ Create New Block".
3. Picking "No Block" sets `RoutineDraft.blockID = nil`. The progression `Picker` in `PrescriptionEditorRow` re-filters to hide `.block` and `.hybrid` cases. Any rows currently set to `.block` or `.hybrid` revert to the routine's previous progression kind (or `.rpe` as fallback) on save.
4. Picking a block sets `RoutineDraft.blockID = block.id`. The progression `Picker` re-filters to show all four kinds. The row label updates to "Block: {block.name}" with accent foreground.
5. Picking "+ Create New Block" pushes `BlockBuilderView` modally. On save-and-dismiss of the builder, the new block's id auto-populates `RoutineDraft.blockID` (the builder returns the saved block via a closure).

### Deload-week banner visibility

1. On `TodayView.appear`, the view computes `PeriodizationEngine.weekContext(for: activeBlock, weekIndex: currentWeekIndex)`.
2. If `weekContext.isDeloadWeek == true`, render `DeloadWeekBanner` at the top of the Today scroll, above `BlockCard`.
3. The banner has no dismiss affordance. It persists for the entire week. It auto-disappears when the user advances into the next (non-deload) week of the block, OR when the user opens Today after `weekContext.weekEndDate < Date.now` (the deload week has ended).
4. The banner background uses the deload phase color tint at 15% opacity. The banner does NOT use the accent color — deload is not an action.

### `ConsiderDeloadBanner` advisory flow (Phase 4 scaffold; Phase 5 fills the signal)

1. On `TodayView.appear`, the view calls `advisory.shouldSuggest(context: currentSessionContext)`. Phase 4 wires this to `StubFatigueAdvisory` which always returns `false`.
2. If `true` (Phase 5+), render `ConsiderDeloadBanner` above `BlockCard` (or above `DeloadWeekBanner` if both are active — though this combination is unlikely since scheduled deloads obviate the advisory).
3. The banner shows the `FatigueSuggestion.reason` as its secondary line.
4. The trailing dismiss button (xmark) sets a per-session dismiss flag in `@State` on `TodayView`. The banner re-evaluates on the next cold app launch.
5. The trailing "Adjust block" CTA opens `BlockBuilderView` for the active block in edit mode. Phase 4 wires the button; Phase 5 fills the action specifics (whether to auto-add a deload phase, or just open the editor).

### Single-active block invariant enforcement

1. `BlockDraft.save(into:context:)` wraps a `try modelContext.transaction { ... }` block.
2. Inside the transaction: if `draft.isActive == true`, execute `for other in try modelContext.fetch(FetchDescriptor<Block>(predicate: #Predicate { $0.isActive && $0.id != draft.id })) { other.isActive = false }` BEFORE materializing the draft.
3. Then create/update the target `Block` from the draft, including `isActive = draft.isActive`.
4. Finally call `try modelContext.save()`.
5. If the transaction throws, the entire change is rolled back — no orphaned "no active block" state can result from a partial save.

### Phase chip kind picker (inside `BlockPhaseEditorRow`)

1. User taps the phase chip on a `BlockPhaseEditorRow`. A `Menu` opens with the four `BlockPhaseKind` options: "Accumulation" / "Intensification" / "Realization" / "Deload".
2. Picking a new kind updates the `BlockPhaseDraft.kind` AND auto-applies that kind's RP-default multipliers (D-10 values) to `volumeMultiplier` and `intensityMultiplier`. The phase chip color updates to the new kind's color.
3. The user can then adjust the multipliers from the defaults if desired. Picking the same kind again is a no-op.

### Snapshot pattern extension (Session.block)

1. User starts a session via `SessionLoggerView` → `Start Workout` button. `SessionFactory.start(routine:on:context:)` is called.
2. The factory snapshots all routine prescription fields onto the new `Session` (Phase 2 pattern). Phase 4 adds one line: `session.block = routine.block`.
3. Even if the user later detaches the routine from the block (sets `routine.block = nil`), the historical session retains `session.block` pointing at the original block context.
4. Phase 6 will query `Session.block` for block-filtered progress views; Phase 4 only writes the snapshot.

The phase-color palette is implemented as a static enum, NOT new asset catalog entries. This keeps the asset catalog free of phase-specific tokens and lets the executor evolve the palette in code-only.

```swift
// fitbod/Periodization/BlockPhaseColors.swift
public enum BlockPhaseColors {
    public static func color(for kind: BlockPhaseKind) -> Color {
        switch kind {
        case .accumulation:    return Color.accentColor              // #0E7C86 / #3FBFC9

        case .deload:          return Color(red: 0.58, green: 0.64, blue: 0.72)  // #94A3B8
        }
    }

    public static func tint(for kind: BlockPhaseKind) -> Color {
        color(for: kind).opacity(0.15)
    }
}
```

This matches the Phase 3 precedent of inline semantic colors for visualization-specific palettes (plate rectangle colors). No new asset catalog entries.

## What's Explicitly Deferred (so the checker doesn't fail it for absence)

These are intentionally NOT in the Phase 4 UI contract — they belong to later phases per `ROADMAP.md` / `04-CONTEXT.md` `<deferred>`:

- **Per-day routine assignment within the mesocycle** (deferred to v1.x) — Phase 4 week pages show a flat list of routines linked to the block; no day-of-week scheduling. The "Routines this week" section header in `MesocycleWeekPage` is intentionally not "Monday: Push A / Tuesday: Pull A" — that's v1.x.
- **Multiple concurrent active blocks** (rejected in CONTEXT.md Q-05) — UI assumes exactly zero or one active block at any time. No UI for "multi-block view."
- **Real fatigue/plateau signal in `ConsiderDeloadBanner`** (Phase 5) — Phase 4 ships the banner view + the `StubFatigueAdvisory` stub. The banner never renders in Phase 4. Phase 5 fills `FatigueAdvisory.shouldSuggest` with the real signal; UI scaffold already locked here.
- **Block-aware progress charting / intent-split charts with block phase shading** (Phase 6) — `BlockCard` shows the active mesocycle; per-exercise time-series charts with phase-tinted background regions are Phase 6. The `Session.block` snapshot (D-20) makes Phase 6 queryable.
- **PRs hit / volume aggregation surfaced inside `BlockReviewView`** (Phase 5 + 6) — Phase 4 ships placeholder copy in the PRs section ("Coming in Phase 6") and computes basic e1RM deltas / total volume inline. Phase 5 fills volume; Phase 6 fills PRs.
- **Block templates marketplace / shareable mesos** (out of scope per `PROJECT.md`) — no social features; personal app only.
- **"Smart" next-phase recommendation engine** — Phase 4 ships static rules (D-23 — Deload→Accumulation, Realization→Deload, Accumulation→Intensification, Intensification→Realization). A historic-fatigue-aware recommendation is a future-version question.
- **Block carryover / "continue this block another N weeks"** (out of scope) — block ends, user starts another (potentially from the same template).
- **Per-phase-kind warm-up scaling beyond deload** (deferred) — Phase 3 handles deload-skip; per-phase variants (e.g., realization = single-set warm-up only) deferred.
- **Block-aware intent overrides** (deferred) — e.g., "intensification automatically biases all `RoutineExercise.intent` to strength." Out of scope; user manages intent at the routine level.
- **Tracking adherence (% of scheduled sessions completed)** (Phase 5 or 6) — not surfaced in Phase 4's `BlockCard` or `BlockReviewView`.
- **Phase-end review surfacing per intermediate phase transition** (rejected in CONTEXT.md D-21 — too noisy) — Phase 4 only triggers `BlockReviewView` at end-of-block, not at end-of-each-phase within a block.
- **Block edit "End block early" overflow action with full mid-block review modal** — Phase 4 wires the menu action and the confirmation alert; the action sets `isActive = false` which then triggers `BlockReviewView` on next Today open via the normal end-of-block flow (so no separate mid-block review modal is needed).
