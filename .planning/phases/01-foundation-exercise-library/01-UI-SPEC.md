# Phase 1 — UI Design Contract

> Visual and interaction contract for Foundation & Exercise Library. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase delivers the first user-visible surfaces in Fitbod: the app's `RootView` with `TabView`, the exercise library (browse / multi-facet filter / type-ahead search), the exercise detail view (read-only built-in + "Copy as custom"), the custom exercise editor (with required primary-muscle stimulus mapping), and the Settings tab (units toggle only). The design stance is **"comprehensive but uncluttered"** — favor information density that a serious lifter expects, but lean on iOS-native chrome (system materials, SF Symbols, Dynamic Type) so the app feels like an iOS app, not a custom design system.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (native SwiftUI — `components.json` / shadcn not applicable to iOS) |
| Preset | not applicable |
| Component library | SwiftUI built-ins (`Form`, `List`, `NavigationStack`, `TabView`, `.searchable`, `.sheet`, `Toggle`, `Slider`, `Picker`) |
| Icon library | SF Symbols (Apple-bundled, weight matches typography) |
| Font | SF Pro — used via Dynamic Type semantic styles, never raw point sizes |

**Why "none":** Per `research/STACK.md` and `PROJECT.md`, the project is locked to zero third-party UI dependencies. Native SwiftUI + SF Symbols + Dynamic Type is the entire toolkit. The shadcn registry / preset model is React-only and not applicable to iOS native.

## Spacing Scale

Declared values (multiples of 4, iOS 8-point grid):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4pt | Icon-to-label gaps inside a row, badge chevron padding |
| sm | 8pt | Compact stack spacing (chip-to-chip in a filter row, filter chip horizontal padding) |
| md | 12pt | List-row vertical padding above/below content, custom-exercise form field gaps |
| lg | 16pt | Default `Form` / `List` horizontal inset, primary content margin from screen edge |
| xl | 24pt | Section spacing inside a long form (between "Identification" / "Muscles" / "Equipment" groups) |
| 2xl | 32pt | Empty-state vertical padding above illustration / message |
| 3xl | 48pt | Top spacer above an empty-state heading when the list is empty on a tall device |

**Exceptions (load-bearing for iOS):**
- **44pt minimum touch target** for any tappable control (HIG-mandated). The filter-chip row uses `8pt` internal padding but the chip's hit area is padded to `44pt` height via `.contentShape(Rectangle()).frame(minHeight: 44)`.
- Custom muscle-weight `Slider` track is 32pt tall to match native iOS slider; thumb hit area is 44pt.
- The "delete" swipe action on a custom-exercise row uses the system-provided swipe height (no override).

## Typography

Use Dynamic Type **semantic styles**, never hard-coded pixel sizes. Mapping below shows the approximate default pt size at the standard `Large` Dynamic Type setting — but the contract is on the semantic role, so users with Accessibility Larger Text get the same hierarchy at scaled-up sizes.

| Role | SwiftUI semantic style | Approx pt @ Large | Weight | Line height | Usage |
|------|------------------------|-------------------|--------|-------------|-------|
| Caption | `.caption` | 12 | regular (400) | 1.3 | Filter-chip count badge, "custom" tag, secondary metadata under exercise name (equipment + mechanic) |
| Body | `.body` | 17 | regular (400) | 1.4 | Exercise name in list rows, form field values, settings labels, primary instructional copy |
| Headline | `.headline` | 17 | semibold (600) | 1.3 | Section headers inside `List` (sectioned by muscle in browse view), custom-exercise editor section titles, navigation bar inline title |
| Title | `.title2` | 22 | semibold (600) | 1.2 | Exercise detail view name at top, empty-state heading, navigation bar large title |

Exactly **4 roles**, exactly **2 weights**. No bold weight (`.heavy` / `.black`) — semibold is the maximum emphasis in this phase.

**Why semantic styles instead of pixels:** the user is a serious lifter; if he ever bumps Dynamic Type to view information-dense screens at arm's length on the gym floor, the hierarchy must scale. Hard-coded pt sizes break this and violate iOS HIG.

## Color

The 60/30/10 split is implemented via **system materials** (auto light/dark mode), plus one designated accent. Specific hex values below describe the **light-mode** appearance; dark-mode values are the iOS-system equivalents auto-applied by the same semantic asset references.

| Role | Light value | Dark value | Source | Usage |
|------|-------------|------------|--------|-------|
| Dominant (60%) | `#FFFFFF` | `#000000` | `Color(.systemBackground)` | Library list background, custom-exercise form background, settings background, root `TabView` content background |
| Secondary (30%) | `#F2F2F7` | `#1C1C1E` | `Color(.systemGroupedBackground)` | Inside grouped `Form` / sectioned `List`, the surface behind list rows; filter-chip default (unselected) fill; tab bar background (iOS auto) |
| Accent (10%) | `#0E7C86` | `#3FBFC9` | `Asset: AccentColor` | (see "Accent reserved for" below) |
| Destructive | `#FF3B30` | `#FF453A` | `Color(.systemRed)` | Delete-custom-exercise confirmation button, custom-exercise validation error text |
| Tertiary text | `#3C3C43` @ 60% opacity | `#EBEBF5` @ 60% opacity | `Color(.secondaryLabel)` | Filter-chip count text, equipment/mechanic metadata under exercise name, settings footer help text |

**Accent color choice:** A deep teal (`#0E7C86` light / `#3FBFC9` dark) instead of stock iOS blue. Rationale: the app's stance is "comprehensive over simple" and "for a serious lifter, not a beginner" — stock `.systemBlue` signals "default iOS app." A deeper teal reads as **precision instrument**, which matches the "intentional prescription" core value. It is still iOS-conventional (within the Apple Maps / Health-app palette range) and clears WCAG AA contrast on both `systemBackground` and `systemGroupedBackground` in both light and dark modes.

**Asset catalog wiring:** This phase populates `Assets.xcassets/AccentColor.colorset/Contents.json` with the teal values (currently empty — confirmed via `Read`). The `fitbod` target's tint then propagates to every `NavigationLink` chevron, `Toggle`, `Slider`, `.searchable` cursor, and selected `TabView` icon automatically. No per-view `.tint(_)` modifiers should be needed.

### Accent reserved for

The accent appears **only** at the following elements in Phase 1. Every other interactive control uses default system styling (`.secondaryLabel` for inactive, `.label` for active text).

1. **Selected filter chip** background fill (unselected chips use `Color(.systemGray5)`).
2. **Selected `TabView` tab icon + label** (system behavior under `.tint(.accentColor)`).
3. **Primary action button** in the custom-exercise editor: "Save Custom Exercise" (filled, accent background, white label).
4. **"Copy as custom"** secondary action in the exercise detail view (text-only, accent foreground, no fill).
5. **Active `Toggle`** in Settings (the lb / kg switch when ON — the system toggle uses tint by default).
6. **Active text cursor + selection** in the `.searchable` field and form text inputs (system behavior).
7. **Stimulus-weight `Slider` track fill** in the custom-exercise muscle-mapping row.

**Explicitly NOT accent:**
- Exercise list row chevron — uses `.secondaryLabel` (system default for `NavigationLink` disclosure).
- Filter-chip count badge — uses `.tertiaryLabel` numeral on the chip's own fill, never accent.
- Section headers in the library — `.headline` on `.label`, never accent.
- Section headers in the custom-exercise editor — `.headline` on `.label`, never accent.

### Stateful helpers (not user-visible but shape the UI contract)

| Type | Role |
|------|------|
| `FilterState` (`@Observable`) | Holds `selectedMuscles: Set<MuscleGroup>`, `selectedEquipment: Set<Equipment>`, `selectedMechanic: Mechanic?`, `selectedPatterns: Set<Pattern>`. Computed `predicate: Predicate<Exercise>`. |
| `CustomExerciseDraft` (`@Observable`) | Mutable draft of a custom exercise — name, mapped-muscles array (each with `weight: Double`), equipment, mechanic, optional `UIImage` reference. `var isValid: Bool` for save-button enablement. |

### Interaction patterns

- **Type-ahead search:** Debounced via `.task(id: searchText)` with a 150ms sleep before re-evaluating the predicate. (Native `.searchable` does not debounce by default; this prevents per-keystroke re-query at 800+ rows.)
- **Filter chip taps:** Each chip presents a `.sheet(detents: [.medium, .large])` listing facet values. Selection is multi-select within a facet, AND across facets. The sheet has its own "Clear" toolbar button for that facet alone.
- **Sectioned list:** Library is sectioned alphabetically by first letter of `canonicalName`. Custom exercises sort in alphabetically (no separate section) — they are visually distinguished by the inline "Custom" tag, not by section grouping.
- **Library list row tap:** Push `ExerciseDetailView` onto the same `NavigationStack`. No modal.
- **Custom exercise "+"** in toolbar: Present `CustomExerciseEditor` as a `.sheet`. On save, pop back to library; the new row appears (via `@Query` reactivity).
- **"Copy as Custom"** on detail view: Initialize a `CustomExerciseDraft` from the current built-in exercise's fields and present `CustomExerciseEditor` as a sheet over the detail view.
- **Tab re-tap pop-to-root:** When the Library tab is tapped while already active, clear its `NavigationPath`. (STACK.md note.)
- **Settings → units toggle:** Two-way bound to `UserSettings.unitsRaw`; flipping it immediately re-renders any view showing weights (n/a in Phase 1 because no weights surface yet — but the contract is in place).

## Accessibility Contract

The user is the only user and is sighted, but iOS HIG accessibility behaviors come "for free" with native SwiftUI and breaking them would be self-sabotage on a gym floor where one-handed / Dynamic Type / VoiceOver scenarios are real.

| Surface | Contract |
|---------|----------|
| Filter chip | `accessibilityLabel` includes facet name + selection count (e.g. "Muscle filter, 2 selected"). |
| Stimulus-weight slider | `accessibilityLabel` = "Stimulus weight for {muscle}". `accessibilityValue` = "{percent}". `accessibilityAdjustableAction` increments/decrements by 0.05. |
| Custom-exercise "Save" button | When disabled, `accessibilityHint` = "Add a primary muscle to enable saving". |
| All icons used as taps without text | Have an `accessibilityLabel` (the toolbar "+" button → "Create custom exercise"). |
| Dynamic Type | All copy uses semantic styles; lay out with `Label`s and `VStack`s that wrap, not fixed-width frames. |
| Color | Accent (#0E7C86 / #3FBFC9) clears WCAG AA against the system backgrounds in both modes. |
| Reduced motion | The filter sheet uses `.sheet` (system-respected). No custom animations introduced this phase. |

## What's Explicitly Deferred (so the checker doesn't fail it for absence)

These are intentionally NOT in the Phase 1 UI contract — they belong to later phases per ROADMAP.md / CONTEXT.md / `<deferred>`:

- **Bundled exercise images / GIFs** — image path strings persist from day 1 but the binary blobs are deferred. The exercise detail view shows no image area in Phase 1 (defer adding the section header).
- **Filter persistence across launches** — filters persist while the Library tab is mounted; iOS TabView preserves tab state by design, so they survive every tab switch. Filters reset only when the app is killed (process termination clears `@State`) or the user explicitly taps "Clear filters". Cross-launch persistence (writing to `UserDefaults` / SwiftData) is the deferred work.
- **Sort options menu** (alphabetical / muscle / equipment) — alphabetical only in v1.
- **Custom-exercise grip / pattern fields** — out of dataset; ROUTINE phase handles deeper exercise metadata.
- **Hand-curated stimulus weights for top-50 compounds** — defaults (1.0 / 0.5) only; curation deferred to Phase 5.
- **"About" rows** in Settings (version, dataset attribution display, etc.) — placeholder header allowed; rows deferred.
- **Plate inventory, smallest increment, per-exercise unit override** — SET-02, SET-03, SET-04 are Phase 3 scope.
- **Today / Routines / Progress tab content** — placeholder views only in Phase 1; built out in Phases 2 / 2 / 6 respectively.
