# Plan 02-01 — RestTimerEngine (Date-based) and Notification Scheduler

> **Wave 2 / Sequence 1.** Ships the canonical mitigation for PITFALLS-doc #4 — the #1 user-visible failure mode if implemented incorrectly. `RestTimerEngine` is a `Date`-based `@Observable` controller; `RestTimerNotificationScheduler` wraps `UNUserNotificationCenter` with the replace-by-identifier reschedule pattern. Tests assert the Date-math against synthetic clocks and prove the same-identifier reschedule is atomic. No ActivityKit integration in this plan (Live Activity lives in plan `02-02`).

## Goal

The rest timer must survive lock-screen + 3min wait without drifting. Per RESEARCH § Pattern 2: the engine stores `startedAt: Date?` + `targetSeconds: Int` and computes `remaining = targetSeconds - Date.now.timeIntervalSince(startedAt)` on read. No foreground `Timer.publish` — that drifts/stops on background. Lock-screen alerting uses `UNUserNotificationCenter.add(request)` with a stable identifier so ±15s mutations are atomic same-identifier replaces.

Splitting the notification logic into `RestTimerNotificationScheduler` (a thin async wrapper) enables unit testing the engine without permission-prompt side effects in the test context.

## Requirements Covered

- **SESS-04** (rest timer: `Date`-based, auto-start on set completion, ±15s, lock-screen `UNUserNotification`, auto-stop on next set entry) — the engine + notification scheduler half. Live Activity / Dynamic Island half lands in plans `02-02` + `02-03`.

### Create — `fitbod/Sessions/RestTimer/RestTimerEngine.swift`

```
import Foundation
import SwiftUI
import UserNotifications

@Observable
@MainActor
public final class RestTimerEngine {
    /// `nil` when the timer is not running. When set, the UI computes
    /// remaining as `Double(targetSeconds) - Date.now.timeIntervalSince(startedAt)`.
    public private(set) var startedAt: Date?
    public private(set) var targetSeconds: Int = 0
    public private(set) var currentExerciseName: String = ""

    /// Stable identifier for the pending local notification. The same
    /// identifier is used on every reschedule — `UNUserNotificationCenter.add`
    /// with a duplicate identifier replaces the prior request, the
    /// documented atomic-reschedule pattern (RESEARCH §6 Pattern 2 / Pitfall 4).
    public static let notificationID = "rest-timer.scheduled"

    /// Injectable scheduler for unit-testability. Production uses
    /// `LiveNotificationScheduler` (UNUserNotificationCenter-backed); tests
    /// pass a stub that records the schedule/cancel calls.
    private let scheduler: RestTimerNotificationScheduling

    /// Injectable clock for unit-testability. `Date.now` in production;
    /// tests pass a closure returning a controlled `Date`.
    private let now: () -> Date

    public init(
        scheduler: RestTimerNotificationScheduling = LiveNotificationScheduler(),
        now: @escaping () -> Date = { Date.now }
    ) {
        self.scheduler = scheduler
        self.now = now
    }

    /// Computed remaining time in seconds. Negative when overrun (the
    /// countdown reached 0 and the user hasn't dismissed yet — UI shows
    /// "Rest complete" until the next set entry).
    public var remaining: TimeInterval {
        guard let startedAt else { return 0 }
        return Double(targetSeconds) - now().timeIntervalSince(startedAt)
    }

    /// Whether the timer is currently active.
    public var isRunning: Bool { startedAt != nil }

    /// Start (or restart) the timer. Called by `SetRow` on completion-checkmark tap.
    /// Schedules a UNUserNotification for the lock-screen alert.
    public func start(seconds: Int, exerciseName: String) {
        self.startedAt = now()
        self.targetSeconds = max(1, seconds)
        self.currentExerciseName = exerciseName
        scheduler.schedule(in: targetSeconds, exerciseName: exerciseName, identifier: Self.notificationID)
    }

    /// Mutate the target. The persisted `startedAt` does NOT move — the
    /// user is asking for more (or less) total rest from the original
    /// start point, not from now.
    /// Per RESEARCH §6 Pitfall 9: the LiveActivity scheduler in plan 02-03
    /// debounces this, but the notification scheduler is always called
    /// here (notification scheduling is cheap and idempotent).
    public func adjust(deltaSeconds: Int) {
        guard isRunning, let startedAt else { return }
        let newTarget = max(0, targetSeconds + deltaSeconds)
        targetSeconds = newTarget

        let firesIn = max(1, Int(startedAt.addingTimeInterval(Double(newTarget)).timeIntervalSince(now())))
        scheduler.schedule(in: firesIn, exerciseName: currentExerciseName, identifier: Self.notificationID)
    }

    /// Stop the timer — called by SetRow when the user taps the NEXT set's
    /// weight field (auto-stop on next-set entry) or by the "Skip" button.
    public func stop() {
        startedAt = nil
        targetSeconds = 0
        currentExerciseName = ""
        scheduler.cancel(identifier: Self.notificationID)
    }
}
```

### Create — `fitbod/Sessions/RestTimer/RestTimerNotificationScheduler.swift`

```
import Foundation
import UserNotifications

/// Abstraction over UNUserNotificationCenter for testability.
@MainActor
public protocol RestTimerNotificationScheduling {
    func schedule(in seconds: Int, exerciseName: String, identifier: String)
    func cancel(identifier: String)
}

/// Production implementation — talks to `UNUserNotificationCenter.current()`.
/// First call to `schedule(...)` triggers the permission prompt (CONTEXT.md
/// Area 4: "request on first session start"). Subsequent calls skip the
/// prompt; denied state silently fails (the in-app overlay still works).
@MainActor
public final class LiveNotificationScheduler: RestTimerNotificationScheduling {
    public init() {}

    public func schedule(in seconds: Int, exerciseName: String, identifier: String) {
        Task { @MainActor in
            let center = UNUserNotificationCenter.current()
            let settings = await center.notificationSettings()

            // Defensive — only schedule if user has granted authorisation.
            // First session start fires the auth prompt; subsequent
            // sessions skip the prompt.
            switch settings.authorizationStatus {
            case .authorized, .provisional, .ephemeral:
                self.addRequest(in: seconds, exerciseName: exerciseName, identifier: identifier, center: center)
            case .notDetermined:
                let granted = (try? await center.requestAuthorization(options: [.alert, .sound])) ?? false
                if granted {
                    self.addRequest(in: seconds, exerciseName: exerciseName, identifier: identifier, center: center)
                }
            case .denied:
                // Silent fallback per UI-SPEC § Error states. Defensive
                // copy lives behind the in-app banner; this function does
                // not surface UI.
                return
            @unknown default:
                return
            }
        }
    }

    public func cancel(identifier: String) {
        UNUserNotificationCenter.current().removePendingNotificationRequests(
            withIdentifiers: [identifier]
        )
    }

    private func addRequest(
        in seconds: Int,
        exerciseName: String,
        identifier: String,
        center: UNUserNotificationCenter
    ) {
        let content = UNMutableNotificationContent()
        content.title = "Rest complete"                                          // UI-SPEC verbatim
        content.body = "\(exerciseName) — next set ready."                       // UI-SPEC verbatim
        content.sound = .default

        // Non-repeating UNTimeIntervalNotificationTrigger has no 60-second
        // minimum (that minimum applies only to repeating triggers per
        // Apple docs). Rest timers commonly run 60-180s but warm-up
        // rests can be 30s. [RESEARCH Assumption A3]
        let trigger = UNTimeIntervalNotificationTrigger(
            timeInterval: max(1, TimeInterval(seconds)),
            repeats: false
        )
        let request = UNNotificationRequest(
            identifier: identifier,
            content: content,
            trigger: trigger
        )
        Task {
            // Adding a request with an existing identifier replaces the
            // previous one — atomic reschedule (RESEARCH §6 Pattern 2 +
            // useyourloaf.com Verified).
            try? await center.add(request)
        }
    }
}
```

### Create — `fitbodTests/RestTimerEngineTests.swift`

```
import Foundation
import Testing
@testable import fitbod

@MainActor
@Suite("RestTimerEngine")
struct RestTimerEngineTests {

    /// In-memory stub used by every test below.
    final class StubScheduler: RestTimerNotificationScheduling {
        var scheduleCalls: [(seconds: Int, name: String, id: String)] = []
        var cancelCalls: [String] = []
        func schedule(in seconds: Int, exerciseName: String, identifier: String) {
            scheduleCalls.append((seconds, exerciseName, identifier))
        }
        func cancel(identifier: String) {
            cancelCalls.append(identifier)
        }
    }

    @Test("notRunningInitially")
    func notRunningInitially() {
        let engine = RestTimerEngine(scheduler: StubScheduler())
        #expect(engine.isRunning == false)
        #expect(engine.remaining == 0)
    }

    @Test("startSetsStartedAtAndTarget")
    func startSetsStartedAtAndTarget() {
        let t0 = Date(timeIntervalSince1970: 1_000)
        var clock = t0
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { clock })

        engine.start(seconds: 180, exerciseName: "Bench Press")
        #expect(engine.isRunning == true)
        #expect(engine.targetSeconds == 180)
        #expect(engine.currentExerciseName == "Bench Press")
        #expect(engine.remaining == 180)

        // After 30 seconds elapse, remaining should be 150.
        clock = t0.addingTimeInterval(30)
        #expect(engine.remaining == 150)

        // After 200 seconds (overrun), remaining goes negative.
        clock = t0.addingTimeInterval(200)
        #expect(engine.remaining == -20)
    }

    @Test("startSchedulesNotificationOnce")
    func startSchedulesNotificationOnce() {
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { Date(timeIntervalSince1970: 1_000) })

        engine.start(seconds: 180, exerciseName: "Bench")
        #expect(stub.scheduleCalls.count == 1)
        #expect(stub.scheduleCalls[0].seconds == 180)
        #expect(stub.scheduleCalls[0].name == "Bench")
        #expect(stub.scheduleCalls[0].id == RestTimerEngine.notificationID)
    }

    @Test("plus15IncreasesTargetAndReschedulesFromOriginalStart")
    func plus15IncreasesTargetAndReschedulesFromOriginalStart() {
        let t0 = Date(timeIntervalSince1970: 1_000)
        var clock = t0
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { clock })

        engine.start(seconds: 180, exerciseName: "Squat")
        clock = t0.addingTimeInterval(30)   // 30s elapsed; 150 remaining
        engine.adjust(deltaSeconds: 15)

        #expect(engine.targetSeconds == 195)
        // remaining = 195 - 30 = 165 (NOT 195 — `startedAt` did NOT move)
        #expect(engine.remaining == 165)
        // Notification rescheduled to fire in (195 - 30) = 165s.
        #expect(stub.scheduleCalls.count == 2)
        #expect(stub.scheduleCalls[1].seconds == 165)
        #expect(stub.scheduleCalls[1].id == RestTimerEngine.notificationID)
    }

    @Test("minus15DecreasesTarget")
    func minus15DecreasesTarget() {
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { Date(timeIntervalSince1970: 1_000) })

        engine.start(seconds: 90, exerciseName: "Curl")
        engine.adjust(deltaSeconds: -15)
        #expect(engine.targetSeconds == 75)
    }

    @Test("adjustClampsTargetAtZero")
    func adjustClampsTargetAtZero() {
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { Date(timeIntervalSince1970: 1_000) })

        engine.start(seconds: 10, exerciseName: "Curl")
        engine.adjust(deltaSeconds: -30)
        #expect(engine.targetSeconds == 0)
    }

    @Test("adjustNoopWhenNotRunning")
    func adjustNoopWhenNotRunning() {
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { Date.now })

        engine.adjust(deltaSeconds: 30)
        #expect(engine.isRunning == false)
        #expect(stub.scheduleCalls.isEmpty)
    }

    @Test("stopCancelsAndResets")
    func stopCancelsAndResets() {
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { Date.now })

        engine.start(seconds: 120, exerciseName: "Row")
        engine.stop()

        #expect(engine.isRunning == false)
        #expect(engine.targetSeconds == 0)
        #expect(engine.currentExerciseName == "")
        #expect(stub.cancelCalls == [RestTimerEngine.notificationID])
    }

    @Test("dateMathSurvivesSimulatedBackground — 3min wait, remaining is correct")
    func dateMathSurvivesSimulatedBackground() {
        let t0 = Date(timeIntervalSince1970: 1_000)
        var clock = t0
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { clock })

        engine.start(seconds: 180, exerciseName: "Bench")
        // Simulate lock + 3min wait. The engine's `startedAt` is fixed
        // at t0; advancing the clock to t0 + 180 → remaining = 0.
        clock = t0.addingTimeInterval(180)
        #expect(engine.remaining == 0)

        // Wait an additional 10 seconds (overrun state).
        clock = t0.addingTimeInterval(190)
        #expect(engine.remaining == -10)
    }

    @Test("restartReplacesPriorState")
    func restartReplacesPriorState() {
        let t0 = Date(timeIntervalSince1970: 1_000)
        var clock = t0
        let stub = StubScheduler()
        let engine = RestTimerEngine(scheduler: stub, now: { clock })

        engine.start(seconds: 120, exerciseName: "Bench")
        clock = t0.addingTimeInterval(60)
        engine.start(seconds: 180, exerciseName: "Squat")  // new set, new exercise

        #expect(engine.targetSeconds == 180)
        #expect(engine.currentExerciseName == "Squat")
        #expect(engine.remaining == 180)
        // Two schedule calls — one per start.
        #expect(stub.scheduleCalls.count == 2)
    }
}
```

## Anti-Patterns to Avoid

- Do NOT use `Timer.publish(every: 1, on: .main)` or any foreground `Timer` to drive the countdown. PITFALLS-doc #4 — the timer drifts/stops on lock. `Date.now`-derived state is the only correct pattern.
- Do NOT call `UNUserNotificationCenter.add(request)` with a unique-per-call identifier (e.g., a UUID). RESEARCH §6 Pitfall 9 — each new identifier creates a separate pending request; ±15s spam would pile up multiple alerts. Use the same `RestTimerEngine.notificationID` constant.
- Do NOT cancel-then-add (two-call). RESEARCH § Standard Stack Alternatives Considered — same-identifier replace-by-identifier is atomic; cancel-then-add has a race window.
- Do NOT move `startedAt` on ±15s. The user is requesting more (or less) total rest from the original start point; persisting `startedAt = .now` would make the timer "jump" the elapsed time backward.
- Do NOT request notification permission at app launch or first-tab tap. CONTEXT.md Area 4: "request on first session start." The `LiveNotificationScheduler.schedule` method handles this — first invocation triggers the auth prompt; later invocations skip.
- Do NOT swallow the permission-denied state without UI feedback. The in-app overlay still works (silent fallback per UI-SPEC § Error states), but the SetRow / SessionLogger surface in plan `04-01` MAY render the defensive "Notifications Disabled" banner. The scheduler itself doesn't surface UI — that belongs to the consuming view.
- Do NOT bake the `LiveNotificationScheduler` directly into `RestTimerEngine.init()` — keep it injectable. The test suite uses a `StubScheduler` to avoid permission prompts in CI.
- Do NOT pre-schedule rest timer notifications at session start (for every planned set). Schedule one notification per active rest period; the next one is scheduled when the next set completes.
- Do NOT introduce any ActivityKit imports in this plan. Plans `02-02` and `02-03` own the Live Activity wiring; this plan ships the engine + notification surface only.
