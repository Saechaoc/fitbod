# Plan 00-01 — Schema V2 New Entities and Additive Fields

> **Wave 0 / Sequence 1.** Adds the three new SwiftData entities (`RoutineFolder`, `SupersetGroup`, `RoutineExerciseSetOverride`) and all additive fields on existing Phase 1 entities (`Routine.folderID`, `RoutineExercise.supersetGroupID` + `tracksTempo` + `tracksPartialReps` + `setOverrides` relationship, `SessionExercise.pinnedNote`, `SetEntry.partialReps` + `clusterSubRepsJoined` + `isComplete`). All changes are additive (new types, new default-valued fields) — eligible for `MigrationStage.lightweight` in plan `00-02`. Closes the data-shape half of ROUTINE-03 / ROUTINE-04 / ROUTINE-06 / SESS-07 / SESS-08 / SESS-11.

## Goal

Land every schema delta Phase 2 needs in a single atomic plan so the schema surface stops moving before any UI plan starts. Three new `@Model` entities + seven additive fields on four existing entities. No SchemaV1 file changes (the new entities are added to SchemaV2 in plan `00-02`); existing entity files get additive fields whose defaults keep V1 stores readable under lightweight migration.

Per RESEARCH § Wave 0 Gaps, this is the canonical "shape the data" plan; every later wave depends on it.

## Requirements Covered

- **ROUTINE-03** (per-set prescription overrides): `RoutineExerciseSetOverride` entity + `RoutineExercise.setOverrides: [RoutineExerciseSetOverride]?` cascade relationship.
- **ROUTINE-04** (supersets / giant sets): `SupersetGroup` entity + `RoutineExercise.supersetGroupID: UUID?` soft ref.
- **ROUTINE-06** (folders): `RoutineFolder` entity + `Routine.folderID: UUID?` soft ref.
- **SESS-07** (optional tempo): `RoutineExercise.tracksTempo: Bool` toggle (SetEntry.tempoActual already exists on Phase 1).
- **SESS-08** (partial reps + cluster sub-reps): `SetEntry.partialReps: Int?` + `SetEntry.clusterSubRepsJoined: String?`; `RoutineExercise.tracksPartialReps: Bool` toggle to surface the partials row in logger UI.
- **SESS-11** (pinned per-exercise notes inline): `SessionExercise.pinnedNote: String?`.
- Plus `SetEntry.isComplete: Bool = false` — the sentinel for "planned but not yet logged" set rows (per RESEARCH Open Question #3 recommendation: prefer explicit Bool over the `completedAt == .distantPast` sentinel; locked here so plan `01-01`'s `SessionFactory.start` can use it).

## Files to Create / Modify

### Create — `fitbod/Models/RoutineFolder.swift`

Per RESEARCH § Code Example 4:

```
import Foundation
import SwiftData

@Model
public final class RoutineFolder {
    @Attribute(.unique) public var id: UUID = UUID()
    public var name: String = ""
    public var sortOrder: Int = 0
    public var createdAt: Date = Date.now

    public init() {}

    public convenience init(name: String, sortOrder: Int = 0) {
        self.init()
        self.name = name
        self.sortOrder = sortOrder
    }
}
```

Per CONTEXT.md Area 6 — single-level folders only (no nesting). `Routine.folderID: UUID?` is a soft ref so deleting a folder never cascades into routines.

### Create — `fitbod/Models/SupersetGroup.swift`

```
import Foundation
import SwiftData

@Model
public final class SupersetGroup {
    @Attribute(.unique) public var id: UUID = UUID()
    public var routineID: UUID = UUID()   // soft ref to source Routine
    public var kindRaw: String = "paired" // "paired" | "giant"
    public var sortOrder: Int = 0
    public var createdAt: Date = Date.now

    public init() {}

    public convenience init(routineID: UUID, kindRaw: String = "paired", sortOrder: Int = 0) {
        self.init()
        self.routineID = routineID
        self.kindRaw = kindRaw
        self.sortOrder = sortOrder
    }
}

public enum SupersetKind: String, CaseIterable, Sendable {
    case paired
    case giant

    public static let `default`: SupersetKind = .paired
}

extension SupersetGroup {
    public var kind: SupersetKind { SupersetKind(rawValue: kindRaw) ?? .paired }
}
```

`routineID` is a soft UUID ref (not a SwiftData relationship). Deleting a routine should cascade its supersets via a query-and-delete pass in the `Routine` deletion path in plan `03-01` (the routine-list view's delete swipe). This keeps the relationship optional in both directions.

### Modify — `fitbod/Models/RoutineExercise.swift`

Add four additive fields below the existing `public var generateWarmups: Bool = false`:

```
public var supersetGroupID: UUID? = nil
public var tracksTempo: Bool = false
public var tracksPartialReps: Bool = false

@Relationship(deleteRule: .cascade, inverse: \RoutineExerciseSetOverride.routineExercise)
public var setOverrides: [RoutineExerciseSetOverride]? = []
```

- `supersetGroupID: UUID?` — soft ref to `SupersetGroup.id`. Deleting a `SupersetGroup` does not cascade; the routine builder zeroes the field via a query-and-null pass.
- `tracksTempo: Bool` — toggles the tempo row in the prescription editor + session logger. Default off (SESS-07: opt-in per exercise).
- `tracksPartialReps: Bool` — toggles the partial-reps row in the session logger. Default off (SESS-08: opt-in per exercise).
- `setOverrides` — cascade relationship; deleting a `RoutineExercise` cascades its per-set overrides.

### Modify — `fitbod/Models/SessionExercise.swift`

Add one line above the `@Relationship(deleteRule: .cascade, inverse: \SetEntry.sessionExercise)` line:

```
public var pinnedNote: String? = nil
```

Per UI-SPEC § Session logger: pinned per-exercise note rendered inline on the session card (icon `pin.fill` + the note text in `.caption` on `Color(.systemYellow).opacity(0.15)` capsule). ### Modify — `fitbod/Models/SetEntry.swift`

Add three additive fields below the existing `public var completedAt: Date = Date.now`:

```
public var partialReps: Int? = nil
public var clusterSubRepsJoined: String? = nil
public var isComplete: Bool = false
```

- `partialReps: Int?` — optional second rep count (e.g. "8 + 2 partials"). Renders only when parent `SessionExercise.routineExercise?.tracksPartialReps == true` (the partials row is opt-in).
- `clusterSubRepsJoined: String?` — comma-joined sub-rep array for rest-pause sets (e.g. `"8,3,2"` represents `[8, 3, 2]`). Computed accessor `clusterSubReps: [Int]` added in the extension below for parsing/joining. Renders only when `setTypeRaw == "rest_pause"`.
- `isComplete: Bool = false` — explicit sentinel that distinguishes "planned but not yet logged" from "logged with zero reps." Per RESEARCH Open Question #3 recommendation: locked here in plan `00-01` so `SessionFactory.start` (plan `01-01`) sets it explicitly. The session logger's set-row completion checkmark flips it to `true`; the UI-SPEC § Per-set completion checkmark contract reads from this flag.

Add a computed accessor in the existing `extension SetEntry { ... }` block:

```
public var clusterSubReps: [Int] {
    get {
        guard let joined = clusterSubRepsJoined, !joined.isEmpty else { return [] }
        return joined.split(separator: ",").compactMap { Int($0) }
    }
    set {
        clusterSubRepsJoined = newValue.isEmpty ? nil : newValue.map(String.init).joined(separator: ",")
    }
}
```

Mirrors the `*Raw: String` + computed-accessor pattern from Phase 1 (FOUND-03). The accessor lives in the existing extension below the `setType` computed property.

## Anti-Patterns to Avoid

- Do NOT add `@Relationship` for `Routine ↔ RoutineFolder` or `RoutineExercise ↔ SupersetGroup` — both are soft `UUID?` refs by design (CONTEXT.md Area 6 + Area 1). A SwiftData relationship would cascade-delete routines when a folder is deleted, which is the opposite of the intended behavior.
- Do NOT use `completedAt == .distantPast` as the "planned but not logged" sentinel — explicit `isComplete: Bool` is cleaner and avoids overloading a `Date` field. RESEARCH Open Question #3 recommendation locked here.
- Do NOT modify `SchemaV1.swift` in this plan. SchemaV1 is the historical baseline and must remain frozen — plan `00-02` introduces `SchemaV2` as the new aggregate (per PITFALLS.md #4 / FOUND-01 / Phase 1's locked schema-versioning stance).
- Do NOT add the new entities to `fitbodApp.swift`'s `Schema(SchemaV1.models)` line — the `Schema()` switch happens in plan `00-02` once `SchemaV2` exists.
- Do NOT inline-parse `clusterSubRepsJoined` in views — the computed `clusterSubReps: [Int]` accessor centralizes the parse/format so the chip row UI in plan `04-02` works against typed `[Int]` not `String`.
