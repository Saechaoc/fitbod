# Plan 00-02 — SchemaV2 VersionedSchema and Lightweight Migration

> **Wave 0 / Sequence 2.** Stands up `SchemaV2: VersionedSchema` aggregating the 12 Phase 1 entities + the 3 new entities created in plan `00-01` (`RoutineFolder`, `SupersetGroup`, `RoutineExerciseSetOverride`). Adds `MigrationStage.lightweight(fromVersion: SchemaV1.self, toVersion: SchemaV2.self)` to `FitbodSchemaMigrationPlan.stages`. Flips `fitbodApp.swift` from `Schema(SchemaV1.models)` to `Schema(SchemaV2.models)`. Adds the canonical lightweight-migration smoke test. This is the project's first schema migration — the load-bearing proof that the Phase 1 versioning scaffold actually works.

## Goal

Open the V1 store under V2 without dropping data. Per RESEARCH § Pattern 4: all schema deltas in plan `00-01` are additive (new entity types + new default-valued fields) — explicitly enumerated as eligible for `MigrationStage.lightweight(fromVersion:toVersion:)` per Apple's docs [CITED: developer.apple.com/documentation/swiftdata/migrationstage/lightweight]. No custom willMigrate/didMigrate closures needed. The test proves V1 stores fixtures round-trip cleanly under V2.

## Requirements Covered

Indirect coverage — this plan ships the migration infrastructure for plan `00-01`'s schema deltas, so it inherits the same requirement IDs (ROUTINE-03, ROUTINE-04, ROUTINE-06, SESS-07, SESS-08, SESS-11) by transitive enablement. The actual user-visible behavior for each requirement lands in later waves.

### Create — `fitbodTests/SchemaV2MigrationTests.swift`

```
import Foundation
import SwiftData
import Testing
@testable import fitbod

@Suite("SchemaV2Migration")
struct SchemaV2MigrationTests {

    @Test("SchemaV2 models list contains every V1 entity plus 3 new types")
    func v2ModelsListIsCompleteSuperset() {
        let v1Names = Set(SchemaV1.models.map { String(describing: $0) })
        let v2Names = Set(SchemaV2.models.map { String(describing: $0) })

        // V1 ⊆ V2
        #expect(v1Names.isSubset(of: v2Names))
        // V2 \ V1 = {RoutineFolder, SupersetGroup, RoutineExerciseSetOverride}
        let added = v2Names.subtracting(v1Names)
        #expect(added == ["RoutineFolder", "SupersetGroup", "RoutineExerciseSetOverride"])
    }

    @Test("FitbodSchemaMigrationPlan registers V1 and V2 and a single lightweight stage")
    func migrationPlanIsWiredCorrectly() {
        let names = FitbodSchemaMigrationPlan.schemas.map { String(describing: $0) }
        #expect(names == ["SchemaV1", "SchemaV2"])
        #expect(FitbodSchemaMigrationPlan.stages.count == 1)
    }

    @Test("Fresh in-memory V2 ModelContainer opens; round-trips a Routine + new entity")
    func freshV2ContainerRoundTripsRoutineAndNewEntities() throws {
        let schema = Schema(SchemaV2.models)
        let config = ModelConfiguration(schema: schema, isStoredInMemoryOnly: true)
        let container = try ModelContainer(
            for: schema,
            migrationPlan: FitbodSchemaMigrationPlan.self,
            configurations: config
        )
        let ctx = ModelContext(container)

        let folder = RoutineFolder(name: "Push / Pull / Legs", sortOrder: 0)
        ctx.insert(folder)

        let routine = Routine()
        routine.name = "Push Day"
        routine.folderID = folder.id           // NEW field
        ctx.insert(routine)

        try ctx.save()

        // Re-fetch and verify the new field survived a save/fetch cycle.
        let fetched = try ctx.fetch(FetchDescriptor<Routine>())
        #expect(fetched.count == 1)
        #expect(fetched.first?.folderID == folder.id)

        let folders = try ctx.fetch(FetchDescriptor<RoutineFolder>())
        #expect(folders.count == 1)
        #expect(folders.first?.name == "Push / Pull / Legs")
    }

    @Test("Cascade: deleting a RoutineExercise cascades into its setOverrides")
    func setOverrideCascadeOnRoutineExerciseDelete() throws {
        let schema = Schema(SchemaV2.models)
        let config = ModelConfiguration(schema: schema, isStoredInMemoryOnly: true)
        let container = try ModelContainer(
            for: schema,
            migrationPlan: FitbodSchemaMigrationPlan.self,
            configurations: config
        )
        let ctx = ModelContext(container)

        let routine = Routine()
        routine.name = "Test"
        ctx.insert(routine)

        let re = RoutineExercise()
        re.routine = routine
        ctx.insert(re)

        let override1 = RoutineExerciseSetOverride(setIndex: 0, targetRepsLow: 6, targetRepsHigh: 6, targetRPE: 8.5)
        override1.routineExercise = re
        let override2 = RoutineExerciseSetOverride(setIndex: 1, targetRepsLow: 8, targetRepsHigh: 10, targetRPE: 7.5)
        override2.routineExercise = re
        ctx.insert(override1)
        ctx.insert(override2)
        try ctx.save()

        #expect(try ctx.fetch(FetchDescriptor<RoutineExerciseSetOverride>()).count == 2)

        ctx.delete(re)
        try ctx.save()

        // RoutineExercise.setOverrides has deleteRule: .cascade → overrides gone.
        #expect(try ctx.fetch(FetchDescriptor<RoutineExerciseSetOverride>()).isEmpty)
    }

    @Test("Soft refs: deleting a RoutineFolder does NOT cascade into routines")
    func folderDeleteDoesNotCascadeIntoRoutines() throws {
        let schema = Schema(SchemaV2.models)
        let config = ModelConfiguration(schema: schema, isStoredInMemoryOnly: true)
        let container = try ModelContainer(
            for: schema,
            migrationPlan: FitbodSchemaMigrationPlan.self,
            configurations: config
        )
        let ctx = ModelContext(container)

        let folder = RoutineFolder(name: "Misc")
        ctx.insert(folder)

        let routine = Routine()
        routine.name = "Stays"
        routine.folderID = folder.id
        ctx.insert(routine)
        try ctx.save()

        ctx.delete(folder)
        try ctx.save()

        // Routine survives; folderID is now dangling (Unfiled re-mapping happens
        // in the folder-delete handler in plan 03-01, not at SwiftData level).
        let routines = try ctx.fetch(FetchDescriptor<Routine>())
        #expect(routines.count == 1)
    }
}
```

Five tests. Together they prove:
1. The V2 schema is a strict additive superset of V1 (the precondition for lightweight migration eligibility).
2. The migration plan is wired with both schemas + a single stage.
3. A fresh V2 container round-trips the new fields and entities.
4. The cascade rule on `RoutineExercise.setOverrides` works (plan 00-01's relationship declaration was correct).
5. The soft-ref design on `Routine.folderID` does NOT cascade routines (Pitfall: a misplaced `@Relationship` would silently break this contract).

Per RESEARCH § Validation Architecture, the lightweight-migration smoke test is owned by Wave 0. Later waves' tests (`SessionFactoryTests`, `RestTimerEngineTests`, etc.) compose on the V2 schema established here.

## Anti-Patterns to Avoid

- Do NOT use `MigrationStage.custom(willMigrate:didMigrate:)` for these deltas. RESEARCH § Pattern 4: additive new-entity adds + additive default-valued field adds are explicitly lightweight-eligible. Adding a custom stage is dead code + extra maintenance + a Pitfall #4 footgun (custom migrations have more failure modes than lightweight).
- Do NOT remove `SchemaV1` from `schemas` array. Even though `migrateV1toV2` chains from V1 → V2, the plan still needs to register V1 as a known historical schema so SwiftData can match an existing on-disk V1 store against the registered version (Hacking with Swift: "all old schema versions must remain registered in the schemas list forever — otherwise upgrade-from-old-versions stops working").
- Do NOT touch `SchemaV1.swift`. SchemaV1 must remain bit-for-bit identical to its Phase 1 form — modifying it would corrupt the migration source-of-truth and is the PITFALLS.md #4 worst-case footgun.
- Do NOT introduce a `SessionLoggerView` Wave 4 plan that opens a private second `ModelContainer` from `Schema(SchemaV1.models)` — every Phase 2 caller goes through the app-scoped V2 container injected via `.modelContainer(container)` in `fitbodApp.swift`.
- Do NOT register `Item.self` (the deleted Xcode template model) in `SchemaV2.models`. PITFALLS.md #4 warning sign: the template `Item` must NEVER appear in any schema.
- Do NOT add `Foundation`-import to `SchemaV2.swift`. Only `import SwiftData` is needed (matches `SchemaV1.swift`).
