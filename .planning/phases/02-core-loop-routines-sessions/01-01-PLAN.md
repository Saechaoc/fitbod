# Plan 01-01 — SessionFactory.start and Previous-Matching-Intent Query

> **Wave 1 / Sequence 1.** Ships the load-bearing snapshot pattern: `SessionFactory.start(routine:on:context:)` deep-copies every prescription field from `RoutineExercise` to `SessionExercise` (and creates planned `SetEntry` rows pre-populated with previous-matching-intent weight hints). Adds the shared `PreviousMatchingIntent` helper that BOTH `SessionFactory.start` (for set-seed weights) AND the future `PreviousColumn` view (plan `04-01`) use to find the most-recent matching-intent prior set. Lands the canonical `SessionFactoryTests` proving editing the routine after `start` leaves the session intact (the PITFALLS-doc #1 / ROUTINE-07 guard). Closes SESS-01 and the data-shape half of ROUTINE-07 + SESS-03.

## Goal

The single source of truth for "start a workout from a routine" — `SessionFactory.start(...)` — is born. Its three jobs are:

1. Insert a new `Session` with `sourceRoutineID: UUID?` soft ref + `routineSnapshotName` copied verbatim.
2. For each `RoutineExercise` in `routine.exercises` (sorted by `orderIndex`), insert a `SessionExercise` snapshotting every prescription field (intent / targetSets / repsLow / repsHigh / RPE / RIR / restSeconds / tempo / progressionKind). Editing the source `RoutineExercise` tomorrow MUST NOT touch this row.
3. Pre-populate planned `SetEntry` rows (one per `targetSets`) seeded with weight from the most recent matching-intent set if available. `SetEntry.isComplete = false` is the explicit "planned but not yet logged" sentinel (plan `00-01` locked this).

The shared `PreviousMatchingIntent` helper is extracted into its own file because BOTH the factory (seeding weights) AND the future `PreviousColumn` view (plan `04-01`'s inline "Previous" column showing `175 × 8 @ 8 (Mon)`) need exactly the same query. Centralizing it now prevents the Phase 2 footgun where the two surfaces diverge in subtle ways.

Per RESEARCH § Pattern 1: the entire `SessionFactory.start` deep-copy is a single `try ctx.save()` transaction so partial state never persists on failure (load-bearing for UI-SPEC § Error states "Couldn't Start Workout" alert).

## Requirements Covered

- **SESS-01** (`SessionFactory.start(...) snapshots all routine prescription fields`): the canonical implementation; proven by `SessionFactoryTests.snapshotsAllPrescriptionFields`.
- **ROUTINE-07** (`editing a routine template never rewrites historical session data`): the snapshot-integrity contract; proven by `SessionFactoryTests.editingRoutineAfterStartLeavesSnapshotIntact`.
- **SESS-03** (`set inputs auto-populate from previous matching-intent session`): the `PreviousMatchingIntent.fetchTopWorkingSet(exerciseID:intentRaw:context:)` helper is the data plumbing; the UI half lands in plan `04-01` (`PreviousColumn` view + auto-populate on tap).

```
import Foundation
import SwiftData

public enum SessionFactoryError: Error {
    case activeSessionAlreadyExists
    case routineHasNoExercises
    case persistenceFailed(underlying: Error)
}

public enum SessionFactory {
    /// Start a new session from a routine. Deep-copies every prescription
    /// field from RoutineExercise to SessionExercise (PITFALLS-doc #1 /
    /// ROUTINE-07) and pre-populates planned SetEntry rows with target
    /// weight pulled from the most recent matching-intent session.
    ///
    /// Returns the newly-inserted Session. Caller is responsible for
    /// presenting `SessionLoggerView(session: returnedSession)`.
    ///
    /// Throws `SessionFactoryError.activeSessionAlreadyExists` if any
    /// `Session.completedAt == nil` row already exists (RESEARCH §6
    /// Pitfall 7). Caller should surface the UI-SPEC § Error states
    /// "Workout in Progress" alert and not invoke this method until the
    /// active session is finished or discarded.
    ///
    /// Throws `SessionFactoryError.routineHasNoExercises` if the routine
    /// has no exercises (defensive — UI-SPEC § Empty states says save is
    /// disabled when `RoutineDraft.isValid == false`, but the factory
    /// guards anyway).
    public static func start(
        routine: Routine,
        on date: Date = .now,
        context: ModelContext
    ) throws -> Session {
        let exercises = (routine.exercises ?? []).sorted { $0.orderIndex < $1.orderIndex }
        guard !exercises.isEmpty else {
            throw SessionFactoryError.routineHasNoExercises
        }

        // RESEARCH §6 Pitfall 7 — enforce one active session at a time.
        let activeDescriptor = FetchDescriptor<Session>(
            predicate: #Predicate { $0.completedAt == nil }
        )
        if let existing = try? context.fetch(activeDescriptor), !existing.isEmpty {
            throw SessionFactoryError.activeSessionAlreadyExists
        }

        let session = Session()
        session.startedAt = date
        session.completedAt = nil
        session.routineSnapshotName = routine.name
        session.sourceRoutineID = routine.id
        session.block = routine.block            // optional relationship; safe to copy
        session.notes = nil
        session.totalDurationSeconds = nil
        context.insert(session)

        for (orderIndex, re) in exercises.enumerated() {
            let se = SessionExercise()
            se.session = session
            se.exercise = re.exercise
            se.orderIndex = orderIndex

            // SNAPSHOT: every prescription field copied. Future edits to
            // `re` will NOT mutate `se` — these are independent rows.
            se.intentRaw = re.intentRaw
            se.targetSets = re.targetSets
            se.targetRepsLow = re.targetRepsLow
            se.targetRepsHigh = re.targetRepsHigh
            se.targetRPE = re.targetRPE
            se.targetRIR = re.targetRIR
            se.prescribedRestSeconds = re.prescribedRestSeconds
            se.tempo = re.tempo
            se.progressionKindRaw = re.progressionKindRaw
            // prescribedWeight is populated by Phase 3's ProgressionStrategy;
            // for Phase 2 we leave it nil and the SetEntry rows pull a
            // weight hint from PreviousMatchingIntent below.
            se.pinnedNote = nil

            context.insert(se)

            // Resolve "previous matching-intent weight" for the planned
            // SetEntry rows. Returns nil for first-ever logged session of
            // this (exercise, intent) tuple → set rows default weight = 0.
            let previousHint = PreviousMatchingIntent.fetchTopWorkingSet(
                exerciseID: re.exercise?.id,
                intentRaw: re.intentRaw,
                context: context
            )?.weight ?? 0

            // Pre-populate planned SetEntry rows. Each set is created
            // with weight from previousHint (a "suggestion") but
            // `isComplete: false` is the explicit "planned, not yet
            // logged" sentinel (plan 00-01 locked this).
            for setIndex in 0..<re.targetSets {
                let entry = SetEntry()
                entry.sessionExercise = se
                entry.orderIndex = setIndex
                entry.weight = previousHint
                entry.reps = 0
                entry.rpe = nil
                entry.setTypeRaw = SetType.working.rawValue
                entry.isWarmup = false
                entry.isComplete = false                 // SENTINEL — set true on completion
                entry.completedAt = date                 // gets overwritten on commit
                context.insert(entry)
            }
        }

        do {
            try context.save()
        } catch {
            throw SessionFactoryError.persistenceFailed(underlying: error)
        }
        return session
    }
}
```

### Create — `fitbod/Sessions/PreviousMatchingIntent.swift`

```
import Foundation
import SwiftData

/// The single shared query for "what did the user log most recently for
/// this exercise at this intent?" Used by:
///   - SessionFactory.start (set-seed weight hint)
///   - PreviousColumn view in plan 04-01 (inline "Previous" column row)
///
/// Centralizing the query prevents the two surfaces from drifting on
/// subtle semantics (working-set filter, warmup exclusion, "top set"
/// tie-breaking).
///
/// Backed by Phase 1's `SessionExercise.intentRaw` #Index for O(log n)
/// lookup. RESEARCH §6 Pitfall 1: extract UUID to local var before
/// constructing the #Predicate (otherwise empty-result footgun on
/// related-entity ID compare).
public struct PreviousMatchingIntentHit: Sendable {
    public let weight: Double
    public let reps: Int
    public let rpe: Double?
    public let sessionStartedAt: Date
}

public enum PreviousMatchingIntent {
    /// Returns the top-working-set from the most recent matching-intent
    /// session for the given (exerciseID, intentRaw) tuple. Returns nil
    /// when no prior matching session exists.
    ///
    /// "Top working set" = highest-weight set among the non-warmup,
    /// reps > 0, isComplete == true entries on the most recent matching
    /// SessionExercise.
    public static func fetchTopWorkingSet(
        exerciseID: UUID?,
        intentRaw: String,
        context: ModelContext
    ) -> PreviousMatchingIntentHit? {
        guard let exerciseID else { return nil }
        let targetID = exerciseID            // local — RESEARCH §6 Pitfall 1
        let targetIntent = intentRaw

        var descriptor = FetchDescriptor<SessionExercise>(
            predicate: #Predicate { se in
                se.intentRaw == targetIntent && se.exercise?.id == targetID
            },
            sortBy: [SortDescriptor(\.session?.startedAt, order: .reverse)]
        )
        descriptor.fetchLimit = 5  // look at last 5 SE rows; first one with
                                   // committed working sets wins
        guard let recent = try? context.fetch(descriptor) else { return nil }

        for se in recent {
            let workingSets = (se.sets ?? [])
                .filter { !$0.isWarmup && $0.reps > 0 && $0.isComplete }
            guard let topSet = workingSets.max(by: { $0.weight < $1.weight }) else { continue }
            return PreviousMatchingIntentHit(
                weight: topSet.weight,
                reps: topSet.reps,
                rpe: topSet.rpe,
                sessionStartedAt: se.session?.startedAt ?? .distantPast
            )
        }
        return nil
    }
}
```

```
import Foundation
```

```
@testable import fitbod

@Suite("PreviousMatchingIntent")
struct PreviousMatchingIntentTests {

    private func makeContext() throws -> ModelContext {
        let schema = Schema(SchemaV2.models)
        let config = ModelConfiguration(schema: schema, isStoredInMemoryOnly: true)
        let container = try ModelContainer(
            for: schema,
            migrationPlan: FitbodSchemaMigrationPlan.self,
            configurations: config
        )
        return ModelContext(container)
    }

    // Creates a completed Session with a single SessionExercise (intent =
    // `intentRaw`, exercise = `ex`) containing the provided set tuples
    // (weight, reps, rpe, isWarmup, isComplete).
    private func makeCompletedSession(
        ctx: ModelContext,
        startedAt: Date,
        exercise: Exercise,
        intentRaw: String,
        sets: [(Double, Int, Double?, Bool, Bool)]
    ) throws -> Session {
        let session = Session()
        session.startedAt = startedAt
        session.completedAt = startedAt.addingTimeInterval(60 * 45)
        session.routineSnapshotName = "Test"
        ctx.insert(session)

        let se = SessionExercise()
        se.session = session
        se.exercise = exercise
        se.intentRaw = intentRaw
        se.targetSets = sets.count
        ctx.insert(se)

        for (i, tup) in sets.enumerated() {
            let entry = SetEntry()
            entry.sessionExercise = se
            entry.orderIndex = i
            entry.weight = tup.0
            entry.reps = tup.1
            entry.rpe = tup.2
            entry.isWarmup = tup.3
            entry.isComplete = tup.4
            entry.completedAt = startedAt
            ctx.insert(entry)
        }
        try ctx.save()
        return session
    }

    @Test("returnsNilWhenNoPriorSession")
    func returnsNilWhenNoPriorSession() throws {
        let ctx = try makeContext()
        let ex = Exercise.previewSample(name: "Bench")
        ctx.insert(ex)
        try ctx.save()

        let hit = PreviousMatchingIntent.fetchTopWorkingSet(
            exerciseID: ex.id,
            intentRaw: Intent.strength.rawValue,
            context: ctx
        )
        #expect(hit == nil)
    }

    @Test("findsTopWorkingSet — excludes warmups and zero-rep sets")
```

## Anti-Patterns to Avoid

- Do NOT have `SessionFactory.start` mutate the source `Routine` (e.g., update a `lastRunAt` timestamp). The factory is read-only against the routine. This was called out in RESEARCH § Anti-Patterns to Avoid.
- Do NOT make `Session.sourceRoutineID` a SwiftData `@Relationship` — it's a soft UUID ref by Phase 1 design (PITFALLS-doc #1 + Pitfall 5). Deleting a routine MUST NOT cascade into history.
- Do NOT use `completedAt == .distantPast` as the planned-vs-logged sentinel — plan `00-01` locked the explicit `isComplete: Bool` instead (RESEARCH Open Question #3).
- Do NOT directly write the matching-intent query inside `SessionFactory.start` body — extract to `PreviousMatchingIntent.fetchTopWorkingSet` (this plan ships the helper) so `PreviousColumn` in plan `04-01` reuses it. Two surfaces, one query.
- Do NOT call `try? ctx.save()` and swallow errors. Use `try ctx.save()` and translate to `SessionFactoryError.persistenceFailed(underlying:)` so the caller can present UI-SPEC's "Couldn't Start Workout" alert.
- Do NOT compare `se.exercise?.id == exerciseID` inside `#Predicate` without extracting `exerciseID` to a local variable first. RESEARCH §6 Pitfall 1 — known SwiftData footgun returns empty results.
- Do NOT skip the `descriptor.fetchLimit = 5` line on `PreviousMatchingIntent`. Without it, the entire matching-intent history loads and the helper would scan thousands of rows after a few months of training data.
- Do NOT call `SessionFactory.start` from the main thread on a `ModelContext` that isn't the main context. The factory uses the caller-provided context — the caller is responsible for thread/actor isolation. (Phase 2 callers always use the main context.)
- Do NOT seed `entry.weight` from a hypothetical Phase 3 `ProgressionStrategy`. Phase 2 only uses `PreviousMatchingIntent` (last session's top working set). Progression-strategy weight prescription is explicitly deferred to Phase 3 per ROADMAP.md.
